# Canonical Release Identity and Evidence Freshness Policy

## Purpose

Every canonical Ordo release must have one identity, one immutable staged tree, and one set of evidence generated by the same delivery-gate run.

## Single source of release identity

`manifests/RELEASE_IDENTITY.json` is authoritative for:

- release ID;
- canonical archive filename;
- internal archive root;
- Ordo Language version;
- Ordo ARF/IRF version.

The release builder must reject an output filename that differs from this manifest. The internal root must never derive from the local checkout directory name.

## Immutable staging and checksum freeze

The builder copies eligible source files into an isolated staging directory. Delivery reports are generated into that staged tree. `SHA256SUMS.txt` is generated last. No canonical file may be changed after checksum generation. Post-checksum mutation invalidates the release.

## Evidence freshness

`reports/delivery/current/DELIVERY_GATE_REPORT.json` and `reports/self-check/current/FINAL_PACKAGE_SELF_CHECK_REPORT.json` must share:

- the same unique run ID;
- the same generation timestamp;
- the same source-tree SHA-256;
- the same release ID.

Historical reports may remain in their archival locations, but root delivery evidence must always belong to the current build run.

## Mandatory post-unpack verification

A release is not canonical until a clean extraction verifies:

1. ZIP structural integrity;
2. every entry in `SHA256SUMS.txt`;
3. archive filename and internal-root identity;
4. release, language, and framework version consistency;
5. gate/self-check run ID, timestamp, and source-tree hash consistency.
