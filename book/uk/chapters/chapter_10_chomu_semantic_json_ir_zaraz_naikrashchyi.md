# Розділ 10. Чому Semantic JSON IR зараз найкращий

## 10.1. Навіщо взагалі потрібен IR

У попередньому розділі ми розібрали, що Ordo Source — це зручна для людини форма опису програми. Її можна читати, обговорювати, редагувати, погоджувати з аналітиком або розробником. Вона схожа на добре структурований YAML-документ: у ній видно intent, contract, state, gates, outputs і handoff.

Але Ordo Source — це ще не найкраща форма для виконання моделлю.

Людині зручно читати так:

```yaml
intent:
  goal: summarize_text

contract:
  output:
    format: bullet_list
    max_items: 3
    language: uk

rules:
  - use_only_input_text
  - do_not_invent_facts
```

Модель теж може це зрозуміти. Але коли процес стає великим, з’являються десятки вузлів, gate-ів, статусів, бібліотек, templates, domain rules і FREEFORM blocks, тоді людський YAML стає занадто вільним. Його можна прочитати по-різному. Частина полів може бути вкладена інакше. Частина правил може бути описана prose-текстом. Частина зв’язків може бути неявною.

Тому потрібен проміжний шар — IR.

![Nebu — ідея: IR як форма виконання](../assets/mascots/64x64/Nebu_idea_64x64.png)

IR означає Internal Representation, тобто внутрішнє представлення програми.

У контексті Ordo IR — це така форма програми, яка вже не призначена в першу чергу для комфортного читання людиною, а призначена для більш точного виконання моделлю або майбутнім Ordo-runtime.

Ordo Source відповідає на питання:

```text
Як людині зручно описати процес?
```

Ordo IR відповідає на питання:

```text
Як моделі або runtime краще виконувати цей процес?
```

Це дуже важлива різниця.

## 10.2. Чому не залишити тільки YAML

Може здатися, що YAML достатній. Він читається, у нього є структура, він набагато кращий за суцільний prompt. Для багатьох простих задач це справді так.

Але YAML лишається authoring-форматом. Він добре підходить для написання й обговорення інструкції. Проте для виконання він має кілька слабких місць.

По-перше, YAML не завжди задає строгий порядок виконання.

Наприклад:

```yaml
steps:
  - read_input
  - extract_points
  - render_summary

gates:
  - input_present
  - no_unsupported_facts
```

Людині зрозуміло, що gates треба застосувати у правильні моменти. Але модель може не завжди зрозуміти, коли саме запускати `input_present`: до `extract_points`, після нього чи перед handoff.

По-друге, YAML часто допускає змішування рівнів. У ньому поруч можуть бути machine-readable поля й довгі пояснення для людини.

По-третє, YAML зручний для читання, але не завжди зручний для перевірки. Runtime-у або слабшій моделі краще мати список конкретних операцій:

```json
[
  {"op": "INTENT.SET", "goal": "summarize_text"},
  {"op": "CONTRACT.DEF", "output": {"format": "bullet_list", "max_items": 3}},
  {"op": "STEP.RUN", "fn": "READ_INPUT_TEXT"},
  {"op": "GATE.CHECK",
  "method": "mechanical",
  "trust_class": "deterministic", "assert": "INPUT_TEXT_PRESENT"}
]
```

Тут уже видно не просто “що є в програмі”, а “що саме треба зробити”.

Саме тому в Ordo Source і IR мають бути різними шарами.

## 10.3. Чому саме Semantic JSON IR

На поточному етапі розвитку AI-моделей найкращим основним IR для Ordo є Semantic JSON IR.

Це означає: JSON-структура, у якій кожна операція має явний `op`, `id`, параметри, входи, виходи, прив’язки й очікувану поведінку.

Наприклад:

```json
{
  "op": "GATE.CHECK",
  "id": "G_NO_UNSUPPORTED_FACTS",
  "method": "self_verification",
  "trust_class": "model_judgment",
  "assert": "NO_UNSUPPORTED_FACTS",
  "source": "RESULT.summary",
  "on_fail": "REPAIR.REMOVE_UNSUPPORTED"
}
```

Це не просто JSON заради JSON. Тут важливо слово Semantic.

Semantic JSON IR означає, що структура не просто містить дані, а прямо несе значення виконання.

`op` каже, яку дію виконати.

`id` дає стабільний ідентифікатор.

`assert` каже, що перевіряється.

`source` каже, до чого застосовується перевірка.

`on_fail` каже, що робити при провалі.

Така форма добре підходить для моделей, тому що вона одночасно:

- достатньо структурована;
- достатньо зрозуміла;
- не занадто стисла;
- не потребує спеціального binary/runtime формату;
- легко перевіряється людиною;
- легко мапиться на майбутній runtime.

## 10.4. Чому не Compact Opcode IR одразу

У попередніх прикладах ми вже бачили компактний формат:

```json
[
  ["I", "summarize_text", "uk"],
  ["C", {"out": {"type": "bullets", "max": 3}}],
  ["G", "NO_UNSUPPORTED_FACTS", "summary", "REPAIR.REMOVE_UNSUPPORTED"]
]
```

Він короткий. Він добре виглядає як майбутній machine-native формат. Але для поточних моделей він ризикований.

Проблема в тому, що компактність зменшує очевидність.

![Nebu — увага: компактність не завжди допомагає](../assets/mascots/64x64/Nebu_attention_64x64.png)

Людина або модель має пам’ятати, що означає `I`, `C`, `G`, який порядок аргументів, які значення допустимі, що робити з помилкою. Для спеціально навченого runtime це нормально. Для звичайної мовної моделі — ще ні.

Compact Opcode IR може стати хорошим майбутнім форматом, якщо:

```text
модель спеціально навчена на Ordo;
або є Ordo execution layer;
або є compiler/runtime, який сам виконує opcodes;
або Compact IR використовується тільки як внутрішній формат, а не як основна інструкція для LLM.
```

Але на практичному етапі краще не поспішати.

Semantic JSON IR — це компроміс між людською зрозумілістю і машинною точністю.

## 10.5. Чому не просто природна мова

Природна мова дуже сильна. Саме вона дозволяє нам пояснити складні речі без формальних схем. Але природна мова має слабке місце: вона часто неоднозначна.

Наприклад:

```text
Не створюй фінальний пакет, якщо контракти не підтверджені.
```

Людині це зрозуміло. Але для виконання виникають питання:

- які саме контракти mandatory;
- що означає confirmed;
- де це перевіряти;
- який статус ставити при провалі;
- чи можна створити draft package;
- чи треба питати користувача;
- чи це hard stop;
- чи це стосується всіх outputs або тільки archive.

У Semantic JSON IR це можна описати точніше:

```json
{
  "op": "GATE.CHECK",
  "id": "G_NO_FINAL_ARCHIVE_BEFORE_CONTRACTS",
  "method": "mechanical",
  "trust_class": "deterministic",
  "assert": "ALL_MANDATORY_CONTRACTS_CONFIRMED",
  "scope": "final_archive",
  "on_fail": "BLOCK_FINAL_ARCHIVE",
  "failed_status": "no_go_requires_confirmation"
}
```

Тут уже менше простору для випадкового тлумачення.

Природна мова залишається важливою. Вона може бути в `FREEFORM`, поясненнях, descriptions, templates. Але hard rules краще виносити в структуровані op-и.

## 10.6. Як виглядає одна й та сама інструкція в трьох формах

Візьмемо просте правило:

```text
Перед фінальним результатом перевір, що немає вигаданих фактів.
```

У природній мові це зрозуміло, але не дуже формально.

В Ordo Source це може виглядати так:

```yaml
gates:
  - id: no_unsupported_facts
    check: no_unsupported_facts
    source: result
    on_fail: remove_or_mark_unsupported
```

У Semantic JSON IR:

```json
{
  "op": "GATE.CHECK",
  "id": "G_NO_UNSUPPORTED_FACTS",
  "method": "self_verification",
  "trust_class": "model_judgment",
  "assert": "NO_UNSUPPORTED_FACTS",
  "source": "RESULT.primary",
  "on_fail": "REPAIR.REMOVE_OR_MARK_UNSUPPORTED"
}
```

У Compact Opcode IR:

```json
["G", "NO_UNSUPPORTED_FACTS", "RESULT.primary", "REPAIR.REMOVE_OR_MARK_UNSUPPORTED"]
```

Усі три форми описують одну ідею. Але вони мають різні ролі.

![Nebu — подумати: одна інструкція у трьох формах](../assets/mascots/64x64/Nebu_thinking_64x64.png)

Природна мова — для пояснення.

Ordo Source — для автора.

Semantic JSON IR — для виконання поточною моделлю або runtime.

Compact Opcode IR — для майбутнього оптимізованого виконання.

## 10.7. Основні переваги Semantic JSON IR

Semantic JSON IR має кілька сильних переваг.

### 1. Явний порядок виконання

IR може бути списком операцій. Це означає, що модель не просто бачить набір правил, а отримує послідовність:

```text
спочатку завантажити документацію;
потім визначити entry;
потім поставити node question;
потім оновити state;
потім перевірити gate;
потім перейти до наступного node.
```

Це особливо важливо для playbook-ів.

### 2. Стабільні ідентифікатори

Кожна операція має `id`.

Це дозволяє посилатися на неї:

```json
{
  "op": "APPROVAL.REQUIRE",
  "id": "A_PASSPORT_APPROVAL",
  "target": "OUTPUT.01_HISTORY_EVENT_PASSPORT"
}
```

Потім у validation report можна сказати:

```json
{
  "approval_id": "A_PASSPORT_APPROVAL",
  "status": "passed"
}
```

Так з’являється traceability.

### 3. Просте логування

Якщо програма виконується покроково, runtime або модель може вести trace:

```json
{
  "executed_op": "G_CONTRACTS_CONFIRMED",
  "status": "failed",
  "reason": "source_field_paths missing",
  "next_action": "ASK_CONFIRMATION"
}
```

Це набагато краще, ніж просто відповідь “потрібно уточнити деякі дані”.

### 4. Легше перевіряти gates

Коли gate описаний як структурований op, його легше перевірити:

```json
{
  "op": "ASSERT.NOT",
  "id": "A_NO_ITEM_PREFIX_IN_DELTA_FIELD",
  "target": "ChangeRecord.delta.field",
  "forbidden_pattern": "^item\\."
}
```

Це вже майже перевірка, яку можна автоматизувати.

### 5. Легше переносити між моделями

Один і той самий Semantic JSON IR можна дати різним моделям. Навіть якщо моделі мають різний стиль відповіді, структура завдання лишається однакова.

Це важливо для майбутнього стандарту Ordo.

## 10.8. Недоліки Semantic JSON IR

Semantic JSON IR не ідеальний.

Його головний недолік — він довший за compact format.

Наприклад, один gate у compact format може займати один рядок, а в Semantic JSON IR — десять рядків.

Але ця довжина купує зрозумілість.

Другий недолік — він потребує дисципліни. Якщо кожен автор буде вигадувати власні op names, IR швидко стане хаотичним. Тому Ordo має мати стандартний каталог op-кодів.

Третій недолік — JSON не дуже приємно писати вручну. Саме тому людина пише Ordo Source, а compiler або модель-помічник переводить його в IR.

Semantic JSON IR — це не обов’язково те, що аналітик має писати руками. Це те, що має отримувати модель для виконання.

## 10.9. Роль compiler-а

У зрілій Ordo-архітектурі між людиною і моделлю має бути compiler або translator.

Людина пише або говорить:

```text
Я хочу, щоб модель вела аналітика через дерево, збирала контракти, не створювала фінальний пакет без підтверджень і перевіряла документи перед архівом.
```

Compiler перетворює це на Ordo Source:

```yaml
entry:
  id: guided_intake

state:
  contracts_confirmed: false
  approvals: pending

gates:
  - no_final_archive_before_contracts
  - rendered_artifacts_validated
```

Потім компілює в Semantic JSON IR:

```json
[
  {"op": "ENTRY.DEF", "id": "guided_intake"},
  {"op": "STATE.SCHEMA", "fields": {"contracts_confirmed": "boolean"}},
  {"op": "GATE.CHECK", "id": "G_NO_FINAL_ARCHIVE",
  "method": "mechanical",
  "trust_class": "deterministic", "assert": "ALL_CONTRACTS_CONFIRMED"}
]
```

У такій схемі модель не мусить сама витягувати весь процес із prose-тексту. Вона отримує вже структуровану карту виконання.

## 10.10. Чому це важливо для слабших моделей

Сильна модель часто може сама здогадатися, що мав на увазі автор. Слабша модель — ні.

Ordo має бути корисною не тільки для найсильнішої моделі. Навпаки, одна з ідей Ordo — зробити складні інструкції виконуваними навіть для моделей, які гірше тримають довгий контекст.

Semantic JSON IR допомагає слабшим моделям, бо:

- зменшує неоднозначність;
- явно задає порядок;
- розділяє операції;
- дає стабільні статуси;
- показує, що робити при fail;
- зменшує потребу “здогадуватись”.

Слабкій моделі простіше виконати:

```json
{"op": "NODE.ASK", "id": "N1", "allowed_answers": ["A", "B", "C"]}
```

ніж довгий текст:

```text
Спочатку, якщо користувач ще не визначив шлях, постав йому перше питання дерева, але не переходь до наступних кроків, доки не отримаєш відповідь...
```

## 10.11. Чому це важливо для майбутнього runtime

Semantic JSON IR також є містком до майбутнього Ordo-runtime.

Runtime може виконувати частину роботи без моделі:

- тримати state;
- визначати current node;
- перевіряти allowed answers;
- запускати gates;
- перевіряти status transitions;
- визначати, які docs треба підключити;
- блокувати фінальний archive;
- формувати validation report.

Модель тоді робить те, що в неї виходить найкраще:

- пояснює користувачу питання;
- інтерпретує відповідь;
- формує зміст документів;
- працює з domain prose;
- пише зрозумілий текст.

Тобто runtime контролює процес, а модель заповнює зміст.

Це важлива стратегічна ідея Ordo.

## 10.12. Практичне правило вибору формату

Для поточної версії Ordo можна зафіксувати таке правило:

```text
Ordo Source — основний формат для людини.
Semantic JSON IR — основний формат для виконання моделлю.
Compact Opcode IR — майбутній формат для оптимізації або native runtime.
FREEFORM — контрольований контейнер для того, що не можна безпечно формалізувати.
```

Якщо ви пишете playbook вручну, починайте з Ordo Source.

Якщо ви даєте інструкцію моделі або runner-у, краще мати Semantic JSON IR.

Якщо у вас з’явиться спеціальний Ordo-runtime, він може компілювати Semantic JSON IR у compact/native форму.

## 10.13. Типові помилки

### Помилка 1. Давати моделі тільки YAML і очікувати runtime-поведінку

YAML допомагає, але він ще не гарантує виконання. Для складних процесів потрібна execution map.

### Помилка 2. Занадто рано переходити на compact opcodes

Компактний формат красивий, але поточним моделям краще давати semantic структуру.

### Помилка 3. Ховати prose-правила всередині description

Якщо правило блокує дію, воно має бути gate або assertion, а не просто текстом.

### Помилка 4. Не мати стабільних id

Без id неможливо нормально робити traceability, validation report і source map.

### Помилка 5. Дозволяти різні значення op для одного й того самого сенсу

Якщо один автор пише `CHECK.GATE`, інший `GATE.CHECK`, а третій `VALIDATE`, стандарт починає розпадатися. Потрібен каталог op-кодів.

## 10.14. Короткий підсумок розділу

Semantic JSON IR зараз найкращий основний формат виконання для Ordo, бо він балансує між зрозумілістю і строгістю.

Він краще за чисту природну мову, бо менше неоднозначний.

Він краще за YAML як execution layer, бо явно задає операції й порядок.

Він практичніший за Compact Opcode IR, бо поточним моделям легше його розуміти.

Головне правило:

```text
Людина пише Ordo Source.
Модель або runtime виконує Semantic JSON IR.
Майбутній native runtime може використовувати Compact Opcode IR.
```

Semantic JSON IR — це міст між людською інструкцією і контрольованим виконанням AI-моделі.

## Міні-вправа

Візьміть просту інструкцію:

```text
Попроси користувача вибрати тип задачі. Якщо він вибрав інцидент, постав питання про критичність. Якщо відповідь не з дозволених варіантів, попроси повторити.
```

Спробуйте записати її у трьох формах:

1. природною мовою;
2. Ordo Source;
3. Semantic JSON IR.

Порівняйте, у якій формі найкраще видно:

- allowed answers;
- current node;
- state update;
- error handling;
- next node.

Якщо у Semantic JSON IR це видно найчіткіше, ви зрозуміли головну ідею цього розділу.
