# Розділ 6.2. Мікропромпти як модульна архітектура процесу

Мікропромпт часто сприймають як коротшу версію великого prompt-а. У цій
книзі він розглядається інакше: як **локальна одиниця поведінки**, яку
можна окремо розвивати, тестувати, версіонувати, замінювати та
відкочувати без перебудови всього процесу.

Головна ідея проста:

> Процес має еволюціонувати як композиція незалежних поведінкових
> модулів, а не як одна велика інструкція.

Це не нормативна вимога мови і не обов’язковий синтаксичний елемент. Це
практична архітектурна рекомендація для процесів, у яких мовна модель
виконує багато різних дій.

## 1. Від великої інструкції до композиції поведінки

### Монолітний підхід

У монолітному підході весь процес описаний одним великим текстом:

    Одна велика інструкція
    ↓
    Будь-яка локальна зміна
    ↓
    Потрібно перечитати і перевірити майже весь процес
    ↓
    Зростає ризик випадково змінити інші частини поведінки

Такий текст може містити правила збору фактів, реконструкції, генерації,
перевірки, підтвердження, маршрути помилок і формат фінальної відповіді.
Навіть якщо змінюється лише одне правило, воно знаходиться всередині
спільного контексту з усіма іншими правилами.

Мовна модель не виконує текст як ізольований набір функцій. Вона
інтерпретує його як єдиний контекст. Через це локальна правка може:

- змінити пріоритет сусідніх інструкцій;
- послабити обмеження іншого кроку;
- додати нове припущення там, де його не було;
- змінити стиль або повноту результатів інших вузлів;
- створити суперечність між початком і кінцем prompt-а;
- збільшити ризик того, що частина правил загубиться в довгому
  контексті.

### Модульний підхід

У модульному підході дерево процесу залишається окремою структурою, а
кожен вузол посилається на власну поведінкову інструкцію:

    Decision Tree
          │
          ├── Step A → Prompt A
          ├── Step B → Prompt B
          ├── Step C → Prompt C
          └── Step D → Prompt D

Якщо потрібно покращити поведінку Step C, змінюється Prompt C. Prompt A,
Prompt B і Prompt D не переписуються без окремої причини.

Це не означає, що інші частини процесу автоматично гарантовано не
зміняться. Але область потенційного впливу стає меншою, видимішою і
легшою для перевірки.

## 2. Що таке мікропромпт у процесній архітектурі

Мікропромпт - це не просто короткий текст. Це локальний поведінковий
модуль, який:

- обслуговує одну визначену відповідальність;
- приймає конкретні входи;
- створює конкретний вихід;
- працює в межах контракту вузла;
- має власні критерії якості;
- може мати власні тести;
- може мати незалежну версію;
- залишає спостережуваний слід виконання.

Корисно розділяти сам вузол і мікропромпт, який реалізує його поведінку:

    Node
       │
       ├── Contract
       ├── Prompt
       ├── Tests
       ├── Trace
       └── Versions

Вузол визначає, **що має бути виконано**. Мікропромпт описує, **як
модель має виконати цю локальну поведінку** в межах дозволеного
контракту.

Тому зміна тексту мікропромпта не повинна непомітно змінювати:

- призначення вузла;
- тип його входу;
- форму його виходу;
- completion criteria;
- дозволені переходи;
- підтверджений стан;
- failure route.

Якщо змінюється хоча б один із цих пунктів, це вже не лише локальний
рефакторинг мікропромпта. Це зміна контракту або архітектури процесу.

## 3. Аналогія з програмною інженерією

Мікропромпти близькі не до окремих абзаців документації, а до модулів
програмної системи.

### Функція

Функція має параметри, виконує одну операцію і повертає результат.
Мікропромпт також має отримувати визначений контекст і створювати один
очікуваний результат.

### Модуль або бібліотека

Модуль приховує внутрішню реалізацію за стабільним інтерфейсом. Так само
текст мікропромпта може змінюватися, поки зберігається контракт вузла.

### Клас

Клас поєднує поведінку з визначеною відповідальністю. Мікропромпт теж
має бути прив’язаний до конкретної ролі, а не намагатися керувати всім
процесом одразу.

### Dependency injection

Процес може посилатися не на жорстко вбудований текст, а на конкретну
версію prompt-модуля. Це дозволяє підставляти іншу реалізацію без зміни
дерева процесу.

### Інтерфейс і ABI/API-контракт

Найважливіша аналогія - стабільний інтерфейс. Безпечна локальна еволюція
можлива лише тоді, коли нова версія мікропромпта не порушує контракт
взаємодії з сусідніми вузлами.

Наприклад:

    node_contract:
      id: reconstruction_step
      input:
        confirmed_facts: list
        open_gaps: list
      output:
        proposed_values: list
        unresolved_gaps: list
        evidence_links: list
      completion_criteria:
        - every proposal has evidence
        - every gap is resolved or explicitly open

Реалізація prompt-а може змінюватися. Але якщо нова версія перестає
повертати `evidence_links`, це вже порушення інтерфейсу.

### Локальний рефакторинг

Рефакторинг покращує внутрішню реалізацію без зміни зовнішньої
поведінки. Аналогічно можна:

- зробити інструкцію точнішою;
- прибрати двозначність;
- додати кращий порядок аналізу;
- уточнити формат локальної відповіді;
- додати захист від типової помилки;

але не змінювати контракт вузла без окремого рішення.

## 4. Основні переваги

### 4.1. Локальність змін

Покращення одного кроку відбувається в одному місці. Автору не потрібно
шукати всі згадки про цю поведінку у великому prompt-і.

Критерій локальності:

> Зміна поведінки одного вузла не вимагає редагування текстів інших
> вузлів, якщо їхні контракти не змінилися.

### 4.2. Менший regression scope

Монолітна інструкція створює широкий regression scope: навіть маленька
правка може вплинути на весь контекст.

    Моноліт
    ↓
    одна локальна зміна
    ↓
    майже повна повторна перевірка процесу

Мікропромпти дозволяють звузити область перевірки:

    Мікропромпт вузла C
    ↓
    локальна зміна
    ↓
    тести вузла C
    ↓
    контрактні тести зв'язків C → наступні вузли
    ↓
    обмежений end-to-end regression

Цифровий відсоток тут не є універсальним. Важливий сам принцип: що
стабільніші контракти між вузлами, то меншу частину процесу потрібно
повторно перевіряти після локальної зміни.

### 4.3. Незалежне версіонування

Кожен мікропромпт може мати власну історію:

    node:
      id: document_review
      contract_version: 1.0.0
      prompt:
        id: review_prompt
        version: 1.4.2
      approved_regression_suite: review-suite-7
      rollback_version: 1.4.1

Важливо розділяти:

11. версію контракту вузла;
12. версію тексту prompt-а;
13. версію тестів;
14. версію процесу, який використовує цей prompt.

Зміна prompt-а не завжди потребує зміни контракту. Але зміна контракту
завжди потребує ширшої перевірки залежностей.

### 4.4. Точкові експерименти

Нову версію можна перевіряти лише на одному вузлі:

- A/B comparison двох версій;
- replay того самого входу;
- порівняння outputs;
- оцінка конкретної метрики;
- перевірка на локальному наборі edge cases.

### 4.5. Простіший review

Рецензент бачить:

- який вузол змінено;
- чому змінено;
- який контракт збережено;
- які тести додано;
- який regression scope перевірено.

Це значно надійніше, ніж переглядати різницю у великій інструкції, де
локальна зміна розчиняється серед десятків інших правил.

### 4.6. Простий rollback

Якщо нова версія погіршила результат, можна повернути попередню версію
одного prompt-модуля без відкату всього процесу.

Rollback безпечний лише тоді, коли попередня версія сумісна з поточним
контрактом вузла.

### 4.7. Кращий execution trace

Trace має фіксувати не тільки назву вузла, а й фактичну версію
поведінки:

    trace_event:
      node_id: reconstruction_step
      contract_version: 1.0.0
      prompt_id: reconstruction_prompt
      prompt_version: 1.4.2
      input_artifact: evidence-104
      output_artifact: proposal-105
      validation_status: passed

Це дозволяє:

- повторити запуск з тією самою версією;
- порівняти результати різних версій;
- локалізувати регресію;
- встановити, яка саме поведінкова реалізація вплинула на результат;
- відтворити історичне рішення.

### 4.8. Локальне накопичення знань

У моноліті нові знання часто додаються в загальний текст. Через деякий
час важко зрозуміти, до якого кроку належить конкретне правило.

У модульній архітектурі досвід зберігається поруч із вузлом:

    Node C
    ├── prompt.md
    ├── contract.yaml
    ├── examples/
    ├── tests/
    ├── regressions/
    ├── known_failures.md
    └── changelog.md

Помилка в конкретному вузлі породжує:

- локальну зміну prompt-а;
- локальний regression test;
- локальну примітку про відому проблему;
- локальне пояснення рішення.

Знання не розмазується по великому документу.

## 5. Абстрактний приклад: реконструкція як локальний модуль

Уявімо процес із чотирьох кроків:

    Збір фактів
    → Реконструкція
    → Генерація документа
    → Перевірка

### Варіант 1. Монолітна інструкція

Уся поведінка описана одним prompt-ом. Після практичного використання
виявилося, що правила реконструкції надто легко приймають правдоподібне
значення за підтверджений факт.

Щоб виправити проблему, автор редагує великий текст. Після цього
потрібно перевірити:

- чи не змінився збір фактів;
- чи не змінився формат reconstruction output;
- чи не почала генерація документа пропускати open gaps;
- чи не змінилась логіка review;
- чи зберігся порядок переходів.

Локальна проблема створила майже повний regression scope.

### Варіант 2. Окремий мікропромпт реконструкції

Процес зберігає стабільну структуру:

    FACT_COLLECTION → RECONSTRUCTION → DOCUMENT_GENERATION → REVIEW

Вузол `RECONSTRUCTION` має окремий prompt. У ньому змінюють лише
правила:

- кожне proposed value має посилання на evidence;
- правдоподібність не дорівнює confirmed;
- невпевнене значення залишається open gap;
- заборонено змінювати confirmed facts;
- output зберігає provenance кожної реконструкції.

Після зміни виконуються:

1.  unit-like тести мікропромпта реконструкції;
2.  контрактний тест його output;
3.  перевірка переходу `RECONSTRUCTION → DOCUMENT_GENERATION`;
4.  короткий end-to-end regression для всього процесу.

Інші prompt-модулі не переписуються.

Критерій успіху:

> Нова версія зменшує помилки реконструкції, зберігає контракт вузла і
> не змінює поведінку сусідніх кроків поза визначеним regression scope.

## 6. Prompt Dependency Graph

Окремі prompt-и не існують у вакуумі. Вони утворюють граф залежностей.

    Process Graph
          │
          ├── Node A ──uses── Prompt A v1.2
          ├── Node B ──uses── Prompt B v2.0
          ├── Node C ──uses── Prompt C v1.4
          └── Node D ──uses── Prompt D v3.1

Але важливіші не лише зв’язки `node → prompt`. Треба бачити контрактні
залежності:

    Prompt A output contract
            ↓
    Prompt B input contract
            ↓
    Prompt C precondition

Тому реєстр мікропромптів має відповідати щонайменше на питання:

- який вузол використовує prompt;
- яка версія активна;
- який контракт вона реалізує;
- які вузли залежать від її output;
- які тести захищають зміну;
- яка версія дозволена для rollback;
- у яких процесах вона використовується повторно.

Умовний реєстр:

    prompt_registry:
      - prompt_id: reconstruction_prompt
        version: 1.4.2
        implements_contract: reconstruction_contract@1.0.0
        used_by:
          - process_a.reconstruction
          - process_b.missing_data_resolution
        tests:
          - reconstruction_unit_suite
          - reconstruction_contract_suite
        rollback_to: 1.4.1

## 7. Контрактна сумісність і межа локальної зміни

Мікропромпти зменшують ризик, але не усувають його.

Локальне оновлення достатнє, якщо:

- призначення вузла не змінюється;
- input schema не змінюється;
- output schema не змінюється;
- semantics статусів не змінюється;
- completion criteria не змінюються;
- failure route не змінюється;
- prompt не отримує нове право змінювати confirmed state.

Локального оновлення недостатньо, якщо:

- вузол починає приймати нове бізнес-рішення;
- змінюється тип output;
- змінюється значення status-а;
- з’являється нова обов’язкова залежність;
- prompt починає виконувати роботу сусіднього вузла;
- змінюється порядок процесу;
- змінюється контракт між вузлами.

У такому випадку потрібні:

- нова версія контракту;
- аналіз Prompt Dependency Graph;
- ширший regression scope;
- оновлення trace semantics;
- можлива міграція процесів, які використовують старий контракт.

## 8. Патерни архітектури мікропромптів

### Pattern: One Node - One Behavioral Module

**Коли застосовувати:** коли вузол має одну чітку відповідальність.

**Структура:**

    Node contract
    → one micro-prompt
    → one observable output
    → local validation

**Перевага:** зміна поведінки легко локалізується.

**Ризик:** prompt може непомітно розростися і почати виконувати кілька
задач.

**Контроль:** перевіряти його разом із критеріями атомарності кроку.

### Pattern: Prompt Behind a Stable Contract

**Коли застосовувати:** коли поведінка часто вдосконалюється, але
інтерфейс має залишатися стабільним.

    Stable node contract
            │
            ├── Prompt v1.2
            ├── Prompt v1.3
            └── Prompt v1.4

**Критерій:** кожна версія проходить однаковий contract suite.

### Pattern: Prompt Registry

**Коли застосовувати:** коли prompt-и повторно використовуються або
мають кілька версій.

Реєстр відділяє посилання на поведінку від тексту процесу. Процес указує
`prompt_id` і version policy, а сам prompt зберігається окремо.

### Pattern: Contract Test Before End-to-End Test

Спочатку перевіряється локальна відповідність контракту, а лише потім
весь процес.

    prompt test
    → output schema test
    → semantic contract test
    → neighboring-node integration
    → selected end-to-end regression

### Pattern: Traceable Prompt Resolution

Перед виконанням runtime або модель повинні зафіксувати, яку версію
prompt-а фактично обрано.

Це захищає від ситуації, коли процес посилається на `latest`, а
історичний запуск неможливо відтворити.

### Pattern: Local Failure Knowledge

Кожна знайдена помилка пов’язується з конкретним prompt-модулем, тестом
і версією виправлення.

    issue:
      id: ISSUE-17
      affected_prompt: reconstruction_prompt@1.4.1
      failure_class: unsupported_inference
      regression_test: REC-CASE-09
      fixed_in: 1.4.2

### Pattern: Approved Promotion

Нова версія не стає активною одразу після редагування.

    draft
    → local tests passed
    → contract tests passed
    → regression passed
    → review approved
    → promoted

## 9. Ризики та антипатерни

### Antipattern: Micro-Prompt in Name Only

Prompt названий мікропромптом, але містить збір даних, рішення,
генерацію, review і state transition.

**Ознака:** зміна однієї частини вимагає перевіряти все всередині
prompt-а.

**Виправлення:** спочатку розділити відповідальності на рівні процесу.

### Antipattern: Hidden Contract in Prompt Text

Контракт існує лише як кілька речень усередині prompt-а.

**Ризик:** нова версія випадково змінює output або статуси.

**Виправлення:** винести contract у явну структуру і тестувати окремо.

### Antipattern: Floating Latest

Процес посилається на «останню» версію без фіксації фактичного
resolution.

**Ризик:** два однакові запуски використовують різну поведінку, а replay
стає недостовірним.

**Виправлення:** фіксувати resolved version в execution trace.

### Antipattern: Shared Prompt with Divergent Responsibilities

Один prompt повторно використовується у вузлах, які мають схожі назви,
але різні контракти.

**Ризик:** покращення для одного процесу погіршує інший.

**Виправлення:** спільне ядро плюс окремі adapters або різні prompt-и.

### Antipattern: Prompt Change Without Regression Artifact

Текст змінено, але немає прикладу, тесту або trace, який показує, яку
проблему виправлено.

**Виправлення:** кожне змістовне покращення має додавати або оновлювати
локальний regression case.

### Antipattern: Prompt Owns the State Transition

Мікропромпт не тільки створює output, а й сам вирішує, що процес тепер
confirmed.

**Ризик:** локальна поведінка отримує право змінювати глобальний стан.

**Виправлення:** state transition залишається окремою операцією з
передумовами.

### Antipattern: Over-Fragmentation

Кожне речення винесене в окремий prompt, хоча дії утворюють одну
механічну операцію.

**Ризик:** надмірна кількість залежностей, складне керування версіями і
зайвий orchestration overhead.

**Виправлення:** межа prompt-модуля має збігатися з реальною межею
відповідальності, а не з довжиною тексту.

## 10. Версіонування та життєвий цикл

Рекомендований мінімальний запис версії:

    micro_prompt:
      id: document_review_prompt
      version: 2.3.0
      implements_contract: document_review@1.1.0
      status: approved
      created_from: 2.2.1
      change_reason: detect missing concrete examples
      regression_suite: review-regression-12
      rollback_to: 2.2.1

Практичне правило зміни версій:

- **patch** - уточнення формулювання без зміни контракту;
- **minor** - нова сумісна поведінка або додатковий локальний захист;
- **major** - несумісна зміна, яка потребує нового контракту або
  міграції залежностей.

Це не жорсткий стандарт, а корисна дисципліна. Головне - щоб історія
дозволяла відповісти:

- що змінилося;
- навіщо;
- яка проблема це виправляє;
- які тести доводять покращення;
- як повернутися назад;
- які процеси використовують цю версію.

## 11. Тестування мікропромптів

### Локальні приклади

Набір коротких inputs, що перевіряють основну поведінку вузла.

### Edge cases

Випадки з прогалинами, суперечностями, неоднозначністю, відсутністю
evidence або неправильним форматом.

### Contract tests

Перевіряють не красу тексту, а дотримання інтерфейсу:

- усі обов’язкові поля присутні;
- статуси мають дозволені значення;
- output не містить заборонених state transitions;
- кожне рішення має evidence;
- open gaps не приховані.

### Metamorphic tests

Перевіряють інваріанти. Наприклад, зміна порядку незалежних фактів не
повинна змінювати рішення.

### Replay tests

Той самий input виконується на старій і новій версіях. Результати
порівнюються за визначеними критеріями.

### Neighbor integration tests

Перевіряється, чи може наступний вузол коректно спожити output нової
версії.

### Selected end-to-end regression

Навіть за стабільного контракту потрібен обмежений наскрізний тест.
Мовні моделі можуть створювати побічні зміни, які формальна схема не
бачить.

## 12. Зв’язок з іншими принципами книги

### Атомарність кроків

Мікропромпт має сенс лише тоді, коли сам вузол атомарний. Якщо вузол
змішує кілька відповідальностей, окремий prompt лише ховає моноліт під
новою назвою.

### Єдина відповідальність

Один prompt-модуль повинен відповідати за один тип поведінки і один
основний результат.

### Явні контракти

Контракт відділяє стабільну зовнішню поведінку від змінної внутрішньої
реалізації.

### Локальні failure routes

Помилка конкретного prompt-а має повертати до найближчого вузла, який
може її виправити, а не перезапускати весь процес.

### Execution trace

Trace фіксує node, prompt ID, prompt version, contract version, input,
output і validation evidence.

### Незалежна валідація

Prompt не повинен сам оголошувати власний результат правильним.
Валідація залишається окремою операцією або окремим вузлом.

### Артефактний підхід

Input і output мікропромпта краще оформлювати як спостережувані проміжні
артефакти. Це робить поведінку тестованою і відтворюваною.

## 13. Практичний checklist

Перед виділенням окремого мікропромпта перевірте:

1.  Чи має вузол одну відповідальність?
2.  Чи існує один основний observable output?
3.  Чи визначені input і output contracts?
4.  Чи можна змінити внутрішню інструкцію без зміни процесу?
5.  Чи має prompt власні тести?
6.  Чи фіксується його версія в trace?
7.  Чи існує локальний rollback?
8.  Чи зрозумілий regression scope?
9.  Чи не змінює prompt confirmed state самостійно?
10. Чи не виконує він приховану роботу сусіднього вузла?
11. Чи не використовується один prompt для несумісних контрактів?
12. Чи є причина для нового prompt-а сильнішою за просте бажання
    скоротити текст?

Перед локальним оновленням перевірте:

- контракт вузла не змінився;
- залежні inputs/outputs сумісні;
- додано regression case;
- локальні тести пройдені;
- contract tests пройдені;
- selected end-to-end regression пройдений;
- resolved version буде записана в execution trace;
- rollback version визначена.

## 15. Підсумкове правило

> Чим менша і чіткіша область відповідальності мікропромпта, тим
> простіше його покращувати, тестувати, відкочувати та повторно
> використовувати.

І ще одне архітектурне узагальнення:

> Стабільний процес повинен залежати від контрактів поведінкових
> модулів, а не від випадкової форми великої інструкції.

Мікропромпти не усувають необхідність системного тестування. Вони
роблять зміни локальнішими, залежності видимішими, regression scope
меншим, а еволюцію процесу - керованою.
