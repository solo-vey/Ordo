# Розділ 37. Ordo без runtime

## Навіщо це потрібно

Коли ми говоримо про Ordo, легко уявити складну систему: compiler, runner, execution engine, test framework, debug console, бібліотеки, coverage reports і повноцінну інфраструктуру. У майбутньому це справді може бути так. Але важливо розуміти: Ordo не починається з runtime.

Ordo може бути корисною ще до появи окремого технічного середовища виконання.

На першому практичному рівні Ordo може працювати як дисциплінований формат інструкцій для моделі. Тобто автор готує Ordo Source або Ordo-подібний документ, передає його моделі, а модель виконує його як структурований execution contract.

У цьому режимі немає зовнішнього runner-а, який технічно блокує неправильні дії. Немає окремого engine, який автоматично рахує coverage або порівнює state snapshots. Але все одно є головне: інструкція вже організована не як довгий prompt, а як керований процес.

Це важливий стартовий режим, бо він дозволяє використовувати Ordo вже зараз.

## Просте пояснення

Ordo без runtime — це коли Ordo-програма існує як структурована інструкція, а роль виконавця бере на себе сама модель.

Модель читає:

```text
- який intent;
- який contract;
- який context;
- який state потрібно вести;
- які nodes проходити;
- які gates перевіряти;
- які outputs створювати;
- де потрібно зупинитися;
- що не можна робити.
```

І після цього намагається виконувати процес згідно з цими правилами.

Тобто Ordo без runtime — це не автоматичне виконання в строгому програмному сенсі. Це контрольоване виконання через мовну дисципліну.

## Чим це відрізняється від звичайного prompt-а

Звичайний prompt часто виглядає так:

```text
Ось велика інструкція. Прочитай її уважно і зроби задачу правильно.
```

Проблема в тому, що модель може не зрозуміти, які частини є обов’язковими, які прикладами, які попередженнями, а які просто поясненнями.

Ordo без runtime формулює інструкцію інакше:

```text
Ось execution contract.
Спочатку визнач intent.
Потім підтвердь contract.
Потім веди state.
Потім рухайся тільки дозволеним path.
Перед output перевір gates.
Якщо gate не пройдений — зупинись.
Якщо правило не формалізоване — використовуй controlled FREEFORM.
Після завершення дай handoff.
```

Навіть без технічного runner-а така структура значно зменшує хаос.

## Мінімальний формат Ordo без runtime

Найпростіша Ordo-програма без runtime може мати такий вигляд:

```yaml
ordo:
  version: "0.11"
  mode: "manual_model_execution"

intent:
  goal: "підготувати аналітичний пакет для нової історичної події"

contract:
  required_confirmations:
    - "alias"
    - "source row"
    - "field mapping"
    - "old/new values"

state:
  fields:
    alias: null
    source_row_confirmed: false
    values_confirmed: false
    package_ready: false

nodes:
  - id: "N1"
    question: "Який alias події?"
    writes_to: "state.alias"

  - id: "N2"
    question: "Яке source field змінюється?"
    requires:
      - "state.alias != null"

outputs:
  - id: "O_FINAL_PACKAGE"
    allowed_when:
      - "G_CONTRACT_CONFIRMED == passed"
      - "G_PRE_ARCHIVE_APPROVAL == passed"

gates:
  - id: "G_PRE_ARCHIVE_APPROVAL"
    type: "blocking"
    rule: "не створювати фінальний пакет без явного підтвердження користувача"
```

Це ще не runtime. Але це вже не хаотичний текст.

## Роль моделі в цьому режимі

У режимі без runtime модель має виконувати одразу кілька ролей:

```text
- читати Ordo Source;
- вести state у своїй відповіді або внутрішньому контексті;
- не переходити через gates без підстав;
- ставити питання за поточним node;
- не генерувати outputs раніше часу;
- пояснювати, де вона знаходиться у процесі;
- фіксувати проблеми, якщо інструкція неоднозначна.
```

Це не ідеально, бо модель усе ще може помилитися. Але це вже набагато краще, ніж сподіватися, що вона “сама правильно зрозуміє” довгу інструкцію.

## Обмеження режиму без runtime

Ordo без runtime має очевидні обмеження.

По-перше, gates не блокуються технічно. Вони блокуються тільки інструкцією. Якщо модель помилиться, вона може перейти далі попри заборону.

По-друге, state не зберігається зовнішньою системою. Він існує в контексті діалогу або в текстовому trace.

По-третє, тести не запускаються автоматично. Їх можна описати, але результат усе одно перевіряє модель або людина.

По-четверте, debug trace залежить від чесності й дисципліни моделі. Якщо модель не покаже причину рішення, зовнішнього механізму примусити її немає.

Тому Ordo без runtime — це хороший старт, але не фінальна форма для складних production-процесів.

## Коли цього достатньо

Режим без runtime може бути достатнім для:

```text
- навчання Ordo;
- написання перших playbook-ів;
- ручного guided intake;
- аналітичних задач з людиною в контурі;
- підготовки документації;
- проектування domain pack-ів;
- попередньої міграції старих інструкцій;
- перевірки концепції перед автоматизацією.
```

Іншими словами, це режим для старту, аналізу і ручного використання.

## Коли runtime уже потрібен

Runtime стає потрібним, коли з’являються високі вимоги до повторюваності, контролю і доказовості.

Наприклад:

```text
- playbook великий і має багато path-ів;
- є критичні blocking gates;
- потрібно гарантувати regression testing;
- outputs використовуються в production;
- є бібліотеки й версії;
- потрібна audit trail;
- потрібно порівнювати різні runs;
- потрібно автоматично збирати improvement records;
- є ризик дорогих помилок.
```

У таких випадках Ordo без runtime уже недостатньо. Потрібен helper-runner або повноцінний execution engine.

## Типові помилки

Перша помилка — думати, що Ordo без runtime вже дає повну детермінованість.

Ні. Це ще не детермінована система. Це структурована домовленість із моделлю.

Друга помилка — не показувати state.

Якщо runtime немає, state потрібно хоча б явно вести в тексті або в компактному status-блоці.

Третя помилка — не відрізняти blocking gates від рекомендацій.

Навіть без runtime потрібно писати:

```text
цей gate блокує подальший рух
```

а не просто:

```text
бажано перевірити
```

Четверта помилка — не мати test cases.

Навіть якщо вони не запускаються автоматично, їх потрібно описати. Інакше автор не знатиме, чи зміна інструкції щось зламала.

П’ята помилка — не збирати feedback.

Якщо користувач постійно вказує на проблеми, але вони не перетворюються на improvement records, Ordo-програма не розвивається системно.

## Міні-вправа

Візьміть одну свою звичайну довгу інструкцію для моделі й спробуйте перевести її в режим Ordo без runtime.

Виділіть:

```text
- intent;
- contract;
- state;
- nodes;
- gates;
- outputs;
- ASSERT.NOT;
- debug information;
- expected tests;
- feedback records.
```

Після цього запитайте себе:

```text
Чи стала інструкція зрозумілішою?
Чи видно, де модель має зупинитися?
Чи видно, що саме потрібно перевірити перед фінальним результатом?
Чи можна буде зрозуміти, чому процес пішов не так?
```

## Короткий підсумок

Ordo без runtime — це перший практичний рівень використання Ordo.

У цьому режимі Ordo ще не виконується спеціальним engine, але вже структурує поведінку моделі через contract, state, nodes, gates, outputs і trace.

Це не заміна runtime, але важливий перехідний формат. Він дозволяє почати використовувати Ordo вже зараз, перевіряти ідеї, мігрувати старі playbook-и й поступово готувати основу для helper-runner або native model support.
