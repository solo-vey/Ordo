# Розділ 6. Entry і Node: як модель ставить питання

## 6.1. Навіщо Ordo потрібні Entry і Node

У звичайному діалозі з моделлю користувач часто очікує, що модель сама зрозуміє, з чого почати.

Наприклад, користувач пише:

```text
Хочу створити нову історичну подію.
```

Модель може відреагувати по-різному. Вона може одразу запропонувати alias. Може попросити source row. Може почати формувати паспорт. Може перейти до QA. Може поставити десять питань одразу.

Для простого діалогу це не завжди критично. Але для playbook-а така свобода небезпечна.

Якщо процес має правильний порядок, модель не повинна сама вирішувати, з якого місця починати. Початок має бути явно визначений.

![Nebu — ідея: Entry як точка входу](../assets/mascots/64x64/Nebu_idea_64x64.png)

У Ordo для цього є `ENTRY.DEF`.

`ENTRY.DEF` відповідає на питання:

```text
З якого місця починається цей процес?
```

А `NODE.DEF` відповідає на питання:

```text
Який конкретний крок або вузол рішення зараз виконується?
```

Разом вони перетворюють діалог із моделлю на керований маршрут.

---

## 6.2. Entry — це двері в процес

`ENTRY.DEF` можна уявити як двері.

Коли користувач каже: “Починаємо”, модель не має бігти по всьому playbook-у. Вона має зайти через визначені двері.

Приклад:

```json
{
  "op": "ENTRY.DEF",
  "id": "START_HISTORY_EVENT_INTAKE",
  "description": "Start guided intake for a new History Event",
  "first_node": "N1_INPUT_TYPE"
}
```

Це означає:

```text
Процес створення історичної події завжди починається з вузла N1_INPUT_TYPE.
```

Не з alias.
Не з display names.
Не з QA.
Не з automation.
Не з повного шаблону паспорта.

А саме з першого питання дерева.

У цьому і є цінність `ENTRY.DEF`: він не дозволяє моделі починати “з того, що їй здається корисним”.

---

## 6.3. Node — це один керований крок

`NODE.DEF` описує один вузол процесу.

Вузол може бути питанням, вибором, перевіркою або проміжним рішенням. У навчальній версії найпростіше думати про node як про одне питання, яке модель ставить користувачу.

Приклад:

```json
{
  "op": "NODE.DEF",
  "id": "N1_INPUT_TYPE",
  "kind": "question",
  "question": "Звідки система має зрозуміти, що історична подія сталася?",
  "answers": ["A", "B", "C", "D", "E", "F"],
  "state_write": "entry_input_type"
}
```

Такий вузол чітко задає:

```text
яке питання ставити;
які відповіді дозволені;
куди записати відповідь;
що робити далі після відповіді.
```

Це важливо, бо модель більше не має вигадувати формат діалогу на ходу.

---

## 6.4. Чому не можна ставити всі питання одразу

![Nebu — увага: не ставити всі питання одразу](../assets/mascots/64x64/Nebu_attention_64x64.png)

Коли модель отримує складне завдання, вона часто намагається бути корисною і питає все одразу:

```text
Надай alias, назву українською та англійською, source row, type/sub_type, field path, values contract, QA fixture, automation scope і runner status.
```

На вигляд це ефективно. Але для користувача це перевантаження. А для процесу — ризик.

Чому?

Тому що частина цих питань ще не має сенсу до вибору шляху.

Наприклад, поки ми не знаємо, чи це Path 1, Path 3, Path 4 або Path 5, неможливо правильно питати про конкретний source contract. Поки не підтверджено source або ChangeRecord, небезпечно питати про `HistoryEvent.item.values`. Поки не підтверджено контракт, рано переходити до QA automation.

Ordo вводить правило:

```text
один вузол — одне головне питання
```

Це не означає, що в процесі буде мало питань. Це означає, що питання йдуть у правильному порядку.

## 6.5. Answer Registry: дозволені відповіді

У звичайному діалозі користувач може відповісти як завгодно. І це добре для природної мови, але погано для керованого процесу.

Якщо вузол очікує `A`, `B`, `C`, `D`, `E`, `F`, а користувач пише `5`, модель має не вгадувати, що він мав на увазі, а перевірити відповідь.

Для цього потрібен `ANSWER.REGISTRY`.

Приклад:

```json
{
  "op": "ANSWER.REGISTRY",
  "node": "N1_INPUT_TYPE",
  "allowed": [
    {
      "key": "A",
      "meaning": "internal_mongo_source_row_change"
    },
    {
      "key": "B",
      "meaning": "new_source_or_data_block_extension"
    },
    {
      "key": "C",
      "meaning": "existing_internal_changerecord"
    },
    {
      "key": "D",
      "meaning": "external_history_fact_or_change_payload"
    },
    {
      "key": "E",
      "meaning": "state_comparison_or_data_block_comparison"
    },
    {
      "key": "F",
      "meaning": "unknown_or_other"
    }
  ],
  "on_invalid": "ASK_VALID_OPTION_AGAIN"
}
```

Це означає, що відповідь користувача має бути перевірена до переходу далі.

Якщо відповідь не входить у allowed list, модель не повинна рухатися наступним вузлом.

---

## 6.6. State update після відповіді

Node не існує сам по собі. Його результат має потрапити в state.

Наприклад, користувач відповів:

```text
A
```

Модель має записати:

```yaml
state:
  entry_input_type: internal_mongo_source_row_change
  path_candidate: Path 1 or Path 2
```

Після цього можна перейти до наступного вузла.

Без state модель може “пам’ятати” відповідь у тексті діалогу, але це ненадійно. У довгому процесі відповідь може загубитися або бути переінтерпретована.

Ordo змушує модель явно оновлювати стан.

---

## 6.7. Node як спосіб не перескакувати вперед

У правильно описаному Ordo-процесі кожен node має межі.

Наприклад, вузол `N1_INPUT_TYPE` має право тільки визначити тип входу. Він не має права підтверджувати alias, не має права вигадувати `type/sub_type`, не має права створювати QA package.

Це можна описати так:

```json
{
  "op": "NODE.DEF",
  "id": "N1_INPUT_TYPE",
  "allowed_actions": [
    "ASK_QUESTION",
    "VALIDATE_ANSWER",
    "STATE.UPDATE",
    "PATH.CANDIDATE.SET"
  ],
  "forbidden_actions": [
    "ALIAS.CONFIRM",
    "SOURCE_ROW.REQUEST_FULL",
    "VALUES.CONTRACT.DEFINE",
    "PACKAGE.GENERATE"
  ]
}
```

Це дуже важлива ідея.

![Nebu — подумати: межі одного node](../assets/mascots/64x64/Nebu_thinking_64x64.png)

Node не тільки каже, що робити. Він також каже, чого на цьому кроці робити не можна.

## 6.8. Приклад маленького дерева

Уявімо простий support-процес.

Користувач пише:

```text
У мене проблема з сервісом.
```

Ordo-процес може починатися так:

```yaml
entry:
  id: START_SUPPORT_TRIAGE
  first_node: N1_REQUEST_TYPE

nodes:
  N1_REQUEST_TYPE:
    question: "Що це за звернення?"
    answers:
      incident: "інцидент / збій"
      change_request: "запит на зміну"
    write_to: request_type

  N2_INCIDENT_SEVERITY:
    when: request_type == incident
    question: "Наскільки критична проблема?"
    answers:
      high: "блокує роботу"
      normal: "не блокує повністю"
    write_to: severity

  N3_INCIDENT_EVIDENCE:
    when: request_type == incident
    question: "Є кроки відтворення або приклад?"
    answers:
      present: "є"
      missing: "немає"
    write_to: reproduction_evidence
```

Це вже дерево рішень. Модель не імпровізує маршрут, а виконує його.

## 6.9. Node і користувацький досвід

Може здатися, що такий підхід робить діалог повільнішим. Але на практиці він часто робить його зручнішим.

Користувач не отримує величезну анкету. Він отримує одне питання за раз.

Модель не вантажить його технічними деталями раніше часу. Вона веде його по маршруту.

Це особливо важливо, коли користувач не знає внутрішніх термінів системи. Наприклад, аналітик не зобов’язаний знати, що таке Path 1 або Path 5. Він може просто відповісти на питання:

```text
Звідки система має зрозуміти, що подія сталася?
```

А модель уже сама мапить відповідь на candidate path.

## 6.10. Типові помилки з Entry і Node

Перша помилка — не мати явного entry.

Тоді модель кожного разу починає по-різному.

Друга помилка — змішати кілька рішень в одному node.

Наприклад:

```text
Оберіть Path, alias, source row і QA fixture.
```

Це поганий node. Він об’єднує кілька різних hard contracts.

Третя помилка — не мати allowed answers.

Якщо відповіді не визначені, модель починає вгадувати.

Четверта помилка — не записувати відповідь у state.

Тоді відповідь є в чаті, але її немає в execution state.

П’ята помилка — дозволити node робити більше, ніж треба.

Наприклад, стартовий node не має права створювати фінальний пакет.

---

## 6.11. Короткий підсумок розділу

`ENTRY.DEF` і `NODE.DEF` потрібні для того, щоб модель не починала процес довільно і не перескакувала між кроками.

`ENTRY.DEF` визначає точку входу.

`NODE.DEF` визначає конкретний вузол процесу.

`ANSWER.REGISTRY` задає дозволені відповіді.

`STATE.UPDATE` зберігає результат відповіді.

Головний принцип:

```text
один вузол — одне головне рішення
```

Це робить діалог керованим, перевірюваним і зручним для користувача.

---

## Міні-вправа

Візьміть будь-який складний процес, де модель має ставити користувачу питання.

Опишіть:

```text
1. Який ENTRY має запускати процес?
2. Який перший NODE?
3. Яке одне головне питання він ставить?
4. Які відповіді дозволені?
5. Куди відповідь записується в STATE?
6. Які дії заборонені на цьому вузлі?
```

Якщо ви не можете відповісти на ці питання, ваш процес ще не готовий до стабільного виконання моделлю.
