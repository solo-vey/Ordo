# Розділ 26. Чому all-in-one більше не головний формат

## Навіщо це потрібно

У багатьох командах складні інструкції для AI-моделі починаються дуже просто: є один великий документ, у якому написано все. Там є правила, приклади, сценарії, винятки, шаблони, заборони, пояснення, таблиці, чеклисти, нотатки, історія змін і підказки для автора.

На початку це зручно. Один файл легко прикріпити до чату. Його легко відкрити. У ньому нібито “все є”. Саме тому формат `all-in-one` часто стає першим природним форматом для playbook-а.

Але коли playbook росте, цей формат починає ламатися.

Проблема не в тому, що великий документ поганий сам по собі. Проблема в тому, що великий документ погано підходить як **головний виконуваний формат** для складного процесу.

Ordo змінює роль `all-in-one`. Він може залишатися зручною зведеною версією для читання, але не має бути єдиним source of truth для виконання.

## Просте пояснення

Уявімо, що у вас є інструкція на 200 сторінок. У ній описано, як модель має створювати аналітичний пакет.

Там є:

```text
- правила вибору сценарію;
- список питань користувачу;
- заборони;
- формати файлів;
- правила тестування;
- приклади;
- edge cases;
- шаблони;
- старі примітки;
- уточнення після помилок;
- винятки;
- чеклисти перед фінальним архівом.
```

Коли модель отримує такий документ як суцільний текст, вона не завжди розуміє, що є процесом, що є прикладом, що є забороною, що є шаблоном, а що є історичною нотаткою.

У великому `all-in-one` дуже легко змішуються різні рівні:

```text
правило + приклад + пояснення + виняток + шаблон + коментар автора
```

Для людини це ще можна читати. Для виконання моделлю це вже небезпечно.

Ordo пропонує іншу модель:

```text
source documents → selected sections → Ordo Source → compiled IR → execution trace → rendered artifacts
```

Тобто великий документ більше не є “тим, що модель просто читає зверху вниз”. Він стає одним із джерел, з якого формуються структуровані execution units.

## Чому all-in-one був корисним

Важливо не відкидати `all-in-one` повністю. Він має свої переваги.

Він корисний для:

```text
- першого збору знань;
- швидкого handoff між людьми;
- оглядового читання;
- збереження повної картини;
- архівної версії;
- експорту в один файл;
- швидкого прикріплення до моделі, якщо немає кращого runtime.
```

На ранній стадії `all-in-one` часто є найпростішим способом зібрати хаотичні знання в одне місце.

Але те, що зручно для збору знань, не завжди зручно для керованого виконання.

## Де all-in-one починає ламатися

Перший тип проблеми — **пошук потрібного правила**.

У великому документі правило може бути написане в одному місці, уточнене в іншому, частково скасоване в третьому і проілюстроване прикладом у четвертому.

Модель може взяти не останнє правило, а найближче за текстом. Або може поєднати старе й нове формулювання.

Другий тип проблеми — **плутанина між правилом і прикладом**.

Наприклад, у документі є приклад події `LU_CHANGE_STATUS`. Модель може випадково сприйняти цей приклад як обов’язковий alias для нового кейсу.

Третій тип проблеми — **пропуск контрольних точок**.

У `all-in-one` може бути написано:

```text
перед фінальним архівом обов’язково виконати self-check
```

Але якщо це просто фраза в довгому тексті, модель може її пропустити. В Ordo це має бути не фраза, а blocking gate.

Четвертий тип проблеми — **складність оновлення**.

Коли користувач каже “це потрібно поправити в інструкції”, незрозуміло, куди саме вносити зміну. У правило? У приклад? У чеклист? У QA? У шаблон? У domain pack? У library?

П’ятий тип проблеми — **неможливість нормального тестування**.

Суцільний markdown-документ важко тестувати. Можна перевірити, чи модель приблизно відповідає правильно, але важко перевірити:

```text
- який path вона обрала;
- який node був активний;
- який gate спрацював;
- який state змінився;
- який FREEFORM блок вплинув на рішення;
- чи не зламався старий сценарій після зміни правила.
```

## Ordo-підхід

В Ordo головним стає не великий документ, а структурована execution model.

`all-in-one` може існувати як rendered artifact, але source of truth для виконання має бути розбитий на керовані частини.

Типова структура може виглядати так:

```text
00_EXECUTION_CONTRACT.md
01_CORE_BINDING.md
02_PROFILE_BINDING.md
03_DOMAIN_PACK.md
04_DECISION_TREE.ordo.yaml
05_OUTPUTS.ordo.yaml
06_GATES.ordo.yaml
07_TESTS.ordo.yaml
08_FREEFORM_LEDGER.md
09_COMPILED_IR.json
10_VALIDATION_REPORT.json
```

Або простіше:

```text
source_docs/
ordo_source/
compiled_ir/
tests/
rendered/
reports/
```

Головна ідея:

```text
Людина може читати зведений документ, але модель має виконувати структуровану карту процесу.
```

## All-in-one як rendered artifact

У новій моделі `all-in-one` не зникає. Він просто змінює статус.

Раніше він був:

```text
main instruction source
```

В Ordo він стає:

```text
rendered human-readable artifact
```

Тобто він може генеруватися з модульних джерел як зручна зведена версія.

Це схоже на технічну документацію: розробники можуть підтримувати багато окремих файлів, а потім збирати один PDF або один HTML-документ для читання.

Ordo має знати, що all-in-one — це не обов’язково первинне джерело. Це може бути результат збірки.

## Чому це важливо для traceability

Коли playbook розбитий на секції, кожне рішення можна прив’язати до конкретного джерела:

```yaml
trace_source: "model_self_report"
knowledge_trace:
  - source: "04_DECISION_TREE.ordo.yaml"
    node: "NODE_SELECT_PATH"
    used_for: "path selection"

  - source: "06_GATES.ordo.yaml"
    gate: "G_PRE_ARCHIVE_APPROVAL"
    used_for: "blocking final archive"
```

У великому `all-in-one` така прив’язка теж можлива, але вона менш точна. Наприклад:

```text
рядки 1420-1480 великого документа
```

Це краще, ніж нічого, але гірше, ніж посилання на конкретний `GATE.DEF`, `NODE.DEF` або `TEST.DEF`.

## Що має робити Documentation Runtime

Коли ми кажемо, що `all-in-one` більше не головний формат, це автоматично веде до наступної ідеї: потрібен Documentation Runtime.

Це буде тема наступного розділу, але тут важливо зафіксувати базову думку.

Documentation Runtime має вміти:

```text
- бачити каталог документів;
- знати, які секції для чого потрібні;
- вибирати тільки релевантні docs для поточного node;
- збирати rendered artifacts;
- перевіряти, що зведений all-in-one не суперечить source files;
- показувати, з якого джерела взяте кожне правило.
```

Тобто замість “прикріпили один величезний файл і сподіваємося, що модель усе правильно прочитає” з’являється керований режим роботи з документацією.

## Ordo-конструкція

У Ordo можна зафіксувати це через такі конструкції:

```text
DOC.CATALOG
DOC.SPLIT
DOC.SELECT
DOC.RENDER
DOC.SOURCE_OF_TRUTH
RENDER.VALIDATE
TRACE.SOURCE
```

Приклад:

```yaml
doc_catalog:
  id: "history_event_playbook_docs"

  source_of_truth:
    mode: "split_docs"
    all_in_one_role: "rendered_artifact"

  documents:
    - id: "execution_contract"
      path: "00_EXECUTION_CONTRACT.md"
      role: "contract"

    - id: "decision_tree"
      path: "04_DECISION_TREE.ordo.yaml"
      role: "path_selection"

    - id: "gates"
      path: "06_GATES.ordo.yaml"
      role: "gate_definitions"

    - id: "tests"
      path: "07_TESTS.ordo.yaml"
      role: "test_suite"
```

А для поточного node:

```yaml
doc_select:
  node: "NODE_SELECT_PATH"
  required_docs:
    - "execution_contract"
    - "decision_tree"
  optional_docs:
    - "domain_examples"
```

## Типова помилка

Погана практика:

```text
У нас є один великий markdown-файл. Нехай модель сама знайде потрібне.
```

Краща практика:

```text
У нас є каталог документів. Для кожного етапу виконання відомо, які секції потрібні, які gates активні, які outputs дозволені, які tests перевіряють поведінку.
```

Ще одна помилка — вважати, що split docs автоматично вирішують проблему.

Розбиття документа саме по собі не дає Ordo. Можна мати 50 маленьких файлів і все одно хаос.

Потрібні ролі, зв’язки, source-of-truth rules і validation.

## Міні-вправа

Візьміть будь-яку велику інструкцію, яку ви використовуєте для AI-моделі.

Спробуйте розділити її не за розміром, а за ролями:

```text
- де описано contract;
- де описано decision tree;
- де описано state;
- де описано gates;
- де описано output;
- де є examples;
- де є controlled FREEFORM;
- де мають бути tests;
- що може бути rendered all-in-one;
- що має бути source of truth.
```

Після цього поставте головне питання:

```text
Чи зможе модель виконувати цей процес без читання всього документа одразу?
```

Якщо відповідь “ні”, значить playbook ще не готовий до Ordo execution.

## Короткий підсумок

`all-in-one` корисний як людський оглядовий або архівний формат, але він погано працює як головний виконуваний формат для складних AI-процесів.

У Ordo source of truth має бути структурованим: contract, nodes, gates, outputs, tests, FREEFORM ledger, libraries і domain rules мають бути виділені в керовані частини.

Зведений `all-in-one` може залишатися, але як rendered artifact, який генерується з модульних джерел і проходить validation.

Головне правило:

```text
У Ordo великий документ можна читати, але виконувати потрібно структуровану модель процесу.
```
