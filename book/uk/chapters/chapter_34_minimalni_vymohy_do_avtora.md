# Розділ 34. Мінімальні вимоги до автора

## Навіщо це потрібно

Ordo дає змогу перетворювати людські інструкції на керований процес для AI-моделі. Але це не означає, що будь-який довгий текст автоматично стає доброю Ordo-програмою. Якщо автор не розуміє, що саме він описує, де процес має зупинятися, які рішення модель може приймати самостійно, а які потребують підтвердження людини, то навіть найкращий синтаксис не врятує інструкцію.

Автор Ordo-програми — це не обов’язково програміст у класичному сенсі. Ним може бути аналітик, методолог, продуктовий спеціаліст, QA, бізнес-експерт або технічний письменник. Але він має мислити не просто текстом, а процесом.

Ordo не вимагає від автора вміння писати код на Java, Python або JavaScript. Проте вона вимагає дисципліни: бачити структуру задачі, відрізняти правило від прикладу, очікуваний результат від побажання, обов’язковий gate від рекомендації, підтверджений факт від припущення.

## Автор Ordo-програми — це проєктувальник поведінки моделі

Коли людина пише звичайний prompt, вона часто формулює тільки бажаний результат:

```text
Підготуй аналітичний пакет для нової історичної події.
```

Для Ordo цього недостатньо. Автор має описати не тільки результат, а й поведінку моделі:

```text
- з чого почати;
- які дані зібрати;
- які питання поставити;
- які варіанти path існують;
- що вважати підтвердженим;
- де модель має зупинитися;
- що заборонено робити без approval;
- які outputs створити;
- які gates виконати перед handoff;
- які debug/test/improvement records мають підтримуватися.
```

Тобто автор Ordo-програми не просто пише інструкцію. Він проєктує керований процес виконання.

## Мінімальна вимога 1. Розуміти intent

Автор має чітко розуміти, для чого існує Ordo-програма.

Погано:

```text
Допомогти аналітику.
```

Краще:

```text
Провести аналітика через guided intake нової історичної події, зібрати підтверджений contract, визначити path, сформувати compact package і виконати validation gates перед handoff.
```

Intent має бути достатньо конкретним, щоб за ним можна було перевірити: програма виконала своє призначення чи ні.

Якщо intent розмитий, далі починаються проблеми: модель ставить не ті питання, переходить не в той path, створює не ті outputs і не розуміє, де треба зупинитися.

## Мінімальна вимога 2. Відрізняти contract від контексту

Одна з типових помилок автора — змішувати обов’язкові умови з фоновим поясненням.

Contract — це те, без чого процес не має права рухатися далі.

Context — це те, що допомагає моделі краще зрозуміти ситуацію.

Наприклад:

```text
Подія стосується зміни статусу компанії.
```

Це може бути context.

А ось це вже contract:

```text
Перед генерацією пакета мають бути підтверджені:
- alias події;
- source row;
- source field;
- old/new values;
- path;
- expected output files.
```

Автор Ordo-програми має явно позначати, що є contract, а що лише поясненням. Якщо цього не зробити, модель може сприйняти важливу умову як необов’язкову пораду.

## Мінімальна вимога 3. Визначати state

Якщо процес має декілька кроків, автор має описати state.

State — це пам’ять процесу: що вже відомо, що підтверджено, що ще очікує рішення, які gates пройдені, які assumptions відкриті.

Без state модель може повторно питати те, що вже було підтверджено, або навпаки — вважати підтвердженим те, що користувач лише обговорював.

Мінімальний state для складної Ordo-програми має містити:

```text
- confirmed facts;
- assumptions;
- selected path;
- pending decisions;
- passed gates;
- blocked gates;
- selected outputs;
- generated artifacts;
- feedback/improvement records.
```

Автор не обов’язково має одразу описувати повну state machine. Але він має хоча б розуміти, які дані процес повинен пам’ятати між кроками.

## Мінімальна вимога 4. Проєктувати nodes як питання, а не як хаотичний діалог

Ordo-програма часто працює як guided intake. Це означає, що модель не просто “розмовляє”, а проходить через вузли процесу.

Кожен node має мати зрозумілу роль:

```text
- зібрати alias;
- визначити path;
- підтвердити source row;
- зібрати values;
- перевірити output contract;
- отримати approval;
- виконати validation;
- сформувати handoff.
```

Поганий node:

```text
Запитай користувача все необхідне.
```

Кращий node:

```text
NODE.DEF collect_source_field:
  goal: "Підтвердити поле джерела, зміна якого створює подію"
  ask: "Яке саме source field є тригером події?"
  required_answer: true
  updates_state:
    - source_field
  next:
    when confirmed: collect_old_new_values
```

Автор має вміти розбивати великий процес на логічні вузли.

## Мінімальна вимога 5. Визначати gates

Gate — це місце, де процес має перевірити умову перед переходом далі.

Автор має вміти відповідати на питання:

```text
Де модель не має права продовжувати без перевірки?
```

Наприклад:

```text
- не створювати фінальний архів без self-check;
- не вважати contract confirmed без явного підтвердження;
- не генерувати code prompt без узгодженого output contract;
- не використовувати source row, якщо він не був підтверджений;
- не створювати HistoryEvent, якщо сценарій є no-op.
```

Якщо автор не визначить gates, модель буде діяти за ймовірністю: іноді правильно, іноді ні.

Ordo потрібна саме для того, щоб такі критичні точки були явно описані.

## Мінімальна вимога 6. Писати негативні правила

Люди часто описують, що модель має зробити, але забувають описати, чого вона не має робити.

Для Ordo це критично.

Потрібні не тільки позитивні інструкції:

```text
Створи QA package.
```

А й негативні:

```text
ASSERT.NOT:
  - do not create final archive before validation passed
  - do not invent missing source values
  - do not mark assumption as confirmed
  - do not hide mandatory gate inside FREEFORM
```

Автор Ordo-програми має вміти бачити небезпечні дії моделі й прямо їх забороняти.

## Мінімальна вимога 7. Визначати outputs

Output — це не просто “відповідь”. У складних процесах output може бути набором документів, JSON-структур, prompt-ів, validation reports, QA packages, changelog entries.

Автор має описати:

```text
- які outputs мають бути створені;
- які outputs є обов’язковими;
- які outputs є допоміжними;
- які поля/секції мають бути всередині;
- які outputs не можна створювати до approval;
- як перевірити, що output готовий.
```

Погано:

```text
Зроби пакет.
```

Краще:

```text
OUTPUT.DEF compact_history_event_package:
  required_files:
    - README.md
    - SUMMARY.json
    - VALIDATION_REPORT.json
    - CONSISTENCY_CHECK_REPORT.json
    - 01_HISTORY_EVENT_PASSPORT_<ALIAS>.md
    - 02_JIRA_TASK_<ALIAS>.md
    - 04_IMPLEMENTATION_PROMPT_<ALIAS>.md
    - 05_QA_PACKAGE_<ALIAS>.md
    - 07_PROCESS_IMPROVEMENT_FEEDBACK_<ALIAS>.md
    - 08_QA_AUTOMATION_SPEC_<ALIAS>.yaml
    - 09_QA_AUTOMATION_README_<ALIAS>.md
```

Output має бути перевірюваним.

## Мінімальна вимога 8. Вміти залишати FREEFORM під контролем

Автор не повинен намагатися формалізувати все. Але він має розуміти, що FREEFORM — це не смітник для складних правил.

FREEFORM можна використовувати для:

```text
- пояснень;
- прикладів;
- доменних нюансів;
- історичних нотаток;
- текстових шаблонів;
- складних людських формулювань.
```

Але не можна ховати в FREEFORM:

```text
- blocking gates;
- required approvals;
- output contracts;
- state transitions;
- status semantics;
- negative assertions.
```

Автор має вміти вирішити: що формалізувати, а що залишити як контрольований FREEFORM.

## Мінімальна вимога 9. Думати про debug і тести з самого початку

Ordo-програма без debug/test layer швидко стає важкою для розвитку.

Автор має хоча б мінімально визначити:

```text
- які paths треба перевірити;
- які gates критичні;
- які сценарії мають бути no-op;
- які помилки вже траплялися;
- які regression tests потрібні;
- які debug logs треба мати для пояснення рішення.
```

Це не означає, що кожна маленька Ordo-програма має мати велику test suite. Але для playbook-ів, domain packs і libraries тести мають бути обов’язковими.

## Мінімальна вимога 10. Фіксувати improvement feedback

Автор Ordo-програми має бути готовий до того, що реальне використання покаже проблеми.

Користувач може сказати:

```text
- це питання треба ставити раніше;
- тут потрібен gate;
- це правило не спрацювало;
- це треба винести в бібліотеку;
- це треба покрити regression test;
- тут FREEFORM треба формалізувати.
```

Усі такі зауваження мають ставати improvement records, а не губитися в чаті.

Автор має підтримувати цикл:

```text
feedback → issue record → affected unit → patch suggestion → test suggestion → approval → regression
```

Саме так Ordo-програма стає кращою після реального використання.

## Що автору не обов’язково вміти

Автору Ordo-програми не обов’язково:

```text
- бути backend-розробником;
- писати compiler;
- знати всі IR op-коди;
- одразу створювати ідеальну структуру;
- формалізувати 100% доменної логіки;
- писати складний runner.
```

Але автор має вміти:

```text
- бачити процес;
- відрізняти правило від прикладу;
- визначати точки зупинки;
- описувати очікувану поведінку;
- фіксувати стан;
- перевіряти outputs;
- приймати feedback і перетворювати його на покращення.
```

## Типові помилки автора

### Помилка 1. Писати Ordo як довгий prompt

Якщо документ виглядає як просто великий текст без nodes, gates, state і outputs, це ще не Ordo-програма.

### Помилка 2. Не визначити, хто приймає рішення

Модель може допомагати, але не всі рішення вона має право приймати сама.

Потрібно явно вказувати:

```text
- model_decision;
- user_approval;
- analyst_decision;
- blocked_until_confirmed.
```

### Помилка 3. Не описати негативні сценарії

Якщо описані тільки успішні сценарії, модель може некоректно діяти в edge cases.

### Помилка 4. Зробити FREEFORM занадто великим

Великий FREEFORM без coverage — це повернення до великого prompt-а.

### Помилка 5. Не передбачити evolution

Ordo-програма майже завжди змінюватиметься. Якщо немає improvement records, changelog і regression tests, кожна зміна буде ризиком.

## Міні-вправа

Візьміть будь-який складний prompt, яким ви вже користувалися, і спробуйте відповісти на питання:

```text
1. Який intent цього prompt-а?
2. Який contract має бути підтверджений перед виконанням?
3. Який state потрібно пам’ятати?
4. Які nodes можна виділити?
5. Де потрібні gates?
6. Які ASSERT.NOT потрібні?
7. Який output має бути створений?
8. Що можна залишити у FREEFORM?
9. Який один debug trace допоміг би зрозуміти помилку?
10. Який один regression test потрібно додати?
```

Якщо ви можете відповісти на ці питання, ви вже мислите як автор Ordo-програми.

## Короткий підсумок

Автор Ordo-програми — це не просто людина, яка пише інструкцію для AI. Це людина, яка проєктує керовану поведінку моделі.

Мінімальні вимоги до автора:

```text
- розуміти intent;
- відрізняти contract від context;
- описувати state;
- будувати nodes;
- визначати gates;
- писати ASSERT.NOT;
- задавати output contract;
- контролювати FREEFORM;
- думати про debug і tests;
- фіксувати improvement feedback.
```

Ordo не вимагає від автора бути програмістом. Але вона вимагає мислити процесом, перевірками і розвитком інструкції.
