# Розділ 14. Чому Ordo-програми потрібно дебажити

## Навіщо це потрібно

Після того як у Ordo зʼявляються `State`, `Node`, `Gate`, `Status semantics` і `Output`, вона перестає бути просто красивим способом писати інструкції. Вона стає програмою поведінки для AI-моделі.

А будь-яку програму потрібно вміти дебажити.

У звичайному програмуванні це очевидно. Якщо код працює не так, розробник дивиться логи, ставить breakpoint, перевіряє значення змінних, дивиться stack trace, повторює запуск на тестових даних.

У роботі з AI-інструкціями часто відбувається навпаки. Ми даємо моделі довгий prompt, отримуємо неправильну поведінку і бачимо тільки фінальну відповідь.

Наприклад:

```text
Модель пропустила важливий крок.
Модель поставила питання не в тому місці.
Модель створила фінальний документ без підтвердження.
Модель переплутала правило і приклад.
Модель обрала неправильний сценарій.
Модель використала старе зауваження як чинне правило.
```

Але головне питання лишається без відповіді:

```text
Чому вона так зробила?
```

Без відповіді на це питання робота з великими інструкціями перетворюється на хаотичне переписування prompt-ів.

Ви виправляєте одну фразу — ламається інша поведінка. Додаєте нове правило — модель починає трактувати його занадто широко. Забороняєте одну дію — модель перестає робити потрібну схожу дію. У результаті зʼявляються десятки версій інструкцій, і вже незрозуміло, яка з них справді краща.

Саме тому debug має бути не зовнішнім допоміжним інструментом, а частиною самої Ordo-мови.

![Nebu — ідея: debug як частина мови](../assets/mascots/64x64/Nebu_idea_64x64.png)

## Просте пояснення

Debug в Ordo — це режим, у якому модель не просто виконує інструкцію, а залишає зрозумілий слід виконання.

Цей слід має відповідати на питання:

```text
- який шлях був обраний;
- які шляхи були відхилені;
- чому був обраний саме цей шлях;
- який state був до кроку;
- як state змінився після кроку;
- який gate спрацював;
- який gate був провалений;
- які правила були використані;
- які знання або документи вплинули на рішення;
- де саме виникла помилка;
- чи була дія дозволена контрактом.
```

У звичайному режимі користувач бачить результат.

У debug-режимі користувач або автор Ordo-програми бачить механіку виконання.

Це різні речі.

Результат може виглядати правильним, але бути отриманим неправильним шляхом. І навпаки: результат може бути неповним, але debug trace покаже, що модель правильно зупинилась на gate, бо їй бракувало підтвердження.

Тому Ordo має оцінювати не тільки фінальний output, а й шлях до нього.

![Nebu — увага: правильний результат може бути отриманий неправильним шляхом](../assets/mascots/64x64/Nebu_attention_64x64.png)

## Чому це особливо важливо для AI

Класична програма виконує інструкції буквально. Якщо в коді написано `if`, програма не розмірковує, чи мав автор на увазі щось інше.

AI-модель працює інакше. Вона інтерпретує інструкції, узагальнює приклади, робить припущення, підставляє контекст, іноді змішує правила з поясненнями.

Це корисно, коли потрібно працювати з людським змістом.

Але це небезпечно, коли потрібно виконати керований процес.

Тому Ordo має дати моделі не тільки інструкцію, а й рамку пояснюваного виконання.

Модель має не просто сказати:

```text
Я вирішила, що треба йти цим шляхом.
```

Вона має показати:

```text
Я обрала Path A1, тому що були підтверджені такі умови.
Я відхилила Path A2, тому що не було підтверджено звʼязану сутність.
Я не перейшла до фінального output, тому що gate G_PRE_APPROVAL ще не passed.
```

Це і є різниця між “модель щось зробила” і “модель виконала Ordo-програму”.

## Ordo-конструкція

Для цього в Ordo вводиться окремий шар:

```text
Ordo Debug, Test & Improvement Layer
```

У цьому розділі нас цікавить перша його частина:

```text
Debug Layer
```

Мінімальні конструкції debug layer:

```text
DEBUG.MODE
TRACE.LOG
DECISION.LOG
PATH.EXPLAIN
STATE.SNAPSHOT
STATE.DIFF
GATE.REPORT
KNOWLEDGE.TRACE
FAILURE.EXPLAIN
RUN.DRY
RUN.REPLAY
```

Їх можна уявити так:

```text
DEBUG.MODE       — вмикає режим пояснюваного виконання;
TRACE.LOG        — загальний журнал виконання;
DECISION.LOG     — журнал прийнятих рішень;
PATH.EXPLAIN     — пояснення вибору або відхилення path;
STATE.SNAPSHOT   — знімок state у певний момент;
STATE.DIFF       — зміна state між двома моментами;
GATE.REPORT      — результат перевірки gates;
KNOWLEDGE.TRACE  — які знання / документи / правила були використані;
FAILURE.EXPLAIN  — пояснення помилки або зупинки;
RUN.DRY          — прогін без реального створення результату;
RUN.REPLAY       — повторне відтворення попереднього запуску.
```

Ці конструкції не є косметичними. Вони потрібні для того, щоб Ordo-програму можна було реально розвивати.

## Малий приклад без debug

Уявімо просту інструкцію:

```text
Підготуй пакет документів, але не створюй фінальний архів без підтвердження користувача.
```

Модель створила фінальний архів одразу.

Без debug ми бачимо тільки проблему:

```text
Фінальний архів створено занадто рано.
```

Але ми не знаємо, чому це сталося.

Можливі причини:

```text
- модель не побачила правило про підтвердження;
- правило було в FREEFORM і не було gate-ом;
- gate існував, але не був blocking;
- state помилково містив approval=true;
- приклад у документі показував фінальний архів і модель сприйняла його як дозвіл;
- інша бібліотека перевизначила поведінку;
- output template був привʼязаний раніше, ніж approval gate.
```

Це різні проблеми. І виправляються вони по-різному.

## Малий приклад з debug

У debug mode Ordo має сформувати приблизно такий trace:

```yaml
run:
  id: "RUN-001"
  mode: "debug"

  trace_source: "model_self_report"
path_selection:
  selected:
    path_id: "package_generation"
    reason: "user requested package creation"

state_snapshots:
  - step: "before_archive_generation"
    state:
      user_approval_received: false
      validation_passed: true

gate_report:
  - gate_id: "G_PRE_ARCHIVE_APPROVAL"
    expected: "passed before archive generation"
    actual: "not_passed"
    severity: "blocking"

violation:
  id: "V_ARCHIVE_CREATED_BEFORE_APPROVAL"
  type: "gate_violation"
  explanation: "final archive generation was attempted before required user approval"
```

Тепер проблема стає зрозумілою.

Це вже не просто “модель помилилась”. Це конкретне порушення:

```text
archive generation attempted before G_PRE_ARCHIVE_APPROVAL passed
```

Таку помилку можна виправляти системно:

```text
- зробити gate blocking;
- додати ASSERT.NOT;
- додати regression test;
- оновити Domain Pack;
- оновити template binding;
- зафіксувати improvement record.
```

## Debug як захист від самовпевнених відповідей

Одна з небезпек AI-моделей — вони можуть звучати впевнено навіть тоді, коли процес був неправильним.

Модель може сказати:

```text
Готово, пакет сформовано.
```

Але debug trace може показати:

```text
- source row не був підтверджений;
- expected values не були зібрані;
- QA package не був перевірений;
- self-check не був виконаний;
- final archive був створений перед approval gate.
```

Тобто Ordo має вчити нас не довіряти тільки гарному фінальному тексту.

Потрібно дивитися, чи був виконаний правильний процес.

## Debug і state

Особливо важливо дебажити `State`.

У складних Ordo-програмах помилки часто виникають не тому, що модель не знає правила, а тому, що вона неправильно вважає state.

Наприклад:

```text
alias_confirmed = true
```

хоча користувач лише запропонував alias, але ще не підтвердив його.

Або:

```text
source_field_known = true
```

хоча поле було взято з припущення, а не з даних.

Тому debug trace має показувати не тільки поточний state, а й причину кожної зміни state.

Погано:

```yaml
state:
  alias_confirmed: true
```

Добре:

```yaml
state_change:
  field: "alias_confirmed"
  before: false
  after: true
  reason: "user explicitly confirmed alias"
  evidence: "message at step 4"
```

Якщо немає reason і evidence, зміна state має бути підозрілою.

## Debug і path selection

Path selection — ще одне критичне місце.

У великих playbook-ах модель часто має обрати один із кількох сценаріїв.

Наприклад:

```text
A1 — зміна основної сутності;
A2 — зміна повʼязаної сутності;
A3 — no-op;
A4 — зовнішня історична подія;
A5 — ручний або нестандартний сценарій.
```

Без debug ми бачимо тільки те, що модель пішла шляхом A1.

З debug ми маємо бачити:

```yaml
selected_path:
  id: "A1"
  reason: "confirmed source row belongs to main entity"

rejected_paths:
  - id: "A2"
    reason: "no related entity identifier confirmed"
  - id: "A4"
    reason: "no ExternalHistoryEvent payload provided"
```

Це дозволяє перевірити не тільки обраний шлях, а й те, що альтернативи були відхилені правильно.

## Debug і знання

Ще одна важлива частина — `KNOWLEDGE.TRACE`.

Модель має показувати, на які джерела вона спиралась:

```text
- Ordo Core;
- Profile;
- Domain Pack;
- Library;
- конкретний rule;
- конкретний template;
- конкретний FREEFORM block;
- user-provided context.
```

Це не означає, що модель має розкривати приватний chain of thought. Ordo debug trace не повинен бути потоком прихованих думок моделі.

Він має бути структурованим звітом про виконання.

Різниця така:

```text
Не потрібно: внутрішні міркування моделі слово в слово.
Потрібно: які правила, дані, gates і state transitions були використані.
```

Ordo debug — це не “прочитати думки моделі”.

Ordo debug — це “побачити протокол виконання”.

## Типові помилки

### Помилка 1. Дебажити тільки фінальний текст

Фінальний текст — це лише результат. У Ordo треба дебажити процес.

### Помилка 2. Не логувати відхилені paths

Важливо знати не тільки що було обрано, а й чому інші варіанти не були обрані.

### Помилка 3. Не логувати state changes

Якщо state змінився без reason/evidence, це потенційна помилка.

### Помилка 4. Вважати debug зайвим для playbook-ів

Навпаки: чим більший playbook, тим більш обовʼязковим є debug.

### Помилка 5. Плутати debug trace з chain of thought

Ordo не потребує приватного потоку думок моделі. Ordo потребує структурованого execution trace.

### Помилка 6. Не зберігати run_id

Без `run_id` неможливо повʼязати помилку, feedback, test і regression.

## Міні-вправа

Візьміть будь-яку складну інструкцію, яку ви вже давали AI-моделі.

Наприклад:

```text
Підготуй задачу для розробника, але спочатку уточни всі невідомі поля і не додавай технічних деталей, яких немає в описі.
```

Спробуйте виписати, що має бути в debug trace:

```text
1. Який intent?
2. Який contract?
3. Які paths можливі?
4. Який path обрано?
5. Які paths відхилено?
6. Які state fields змінюються?
7. Які gates мають бути blocking?
8. Які outputs заборонені до підтвердження?
9. Які знання або документи використані?
10. Який failure мав би бути показаний, якщо модель пропустить уточнення?
```

Після цього спробуйте сформулювати один gate, який обовʼязково має зʼявитись у trace.

Наприклад:

```text
G_UNKNOWN_FIELDS_RESOLVED
```

І одне порушення:

```text
V_TASK_CREATED_WITH_UNCONFIRMED_FIELD
```

## Короткий підсумок

Ordo-програму потрібно дебажити, бо вона керує поведінкою AI-моделі, а не просто описує бажаний текст.

Debug Layer потрібен, щоб бачити:

```text
- вибір path;
- відхилення альтернатив;
- зміни state;
- оцінку gates;
- використані знання;
- порушення;
- причину зупинки;
- причину помилки.
```

Головне правило:

```text
У складній Ordo-програмі неправильний результат — це не просто неправильний output, а неправильний або непояснений execution path.
```

Тому Ordo має підтримувати debug як частину мови.

Без debug Ordo-процеси неможливо надійно розвивати.

З debug Ordo стає не тільки мовою виконання, а й мовою пояснюваного виконання.
