# Розділ 3. Ordo як мова керування моделлю

## 3.1. Не просто формат, а спосіб керування поведінкою

Коли люди вперше бачать Ordo, вони часто питають:

```text
То це YAML? JSON? Набір op-кодів? Новий формат prompt-ів?
```

Правильна відповідь: Ordo може використовувати YAML, JSON або компактний IR, але сама Ordo не зводиться до жодного з цих форматів.

YAML — це спосіб записати Ordo Source так, щоб людині було зручно читати.

JSON — це спосіб передати структуровану інструкцію моделі або runtime.

Compact IR — це майбутній спосіб зробити інструкції коротшими й ближчими до машинного виконання.

Але Ordo як ідея — це не формат. Це мова керування поведінкою моделі.

<p><img src="../assets/mascots/64x64/Nebu_idea_64x64.png" alt="Nebu idea" width="64" /></p>

Її задача — не просто красиво записати інструкцію, а зробити так, щоб модель:

```text
- знала, з чого почати;
- розуміла, який крок виконує зараз;
- не перескакувала вперед;
- зберігала state;
- відрізняла підтверджене від припущеного;
- виконувала gates;
- зупинялася, якщо результат ще не можна фіналізувати;
- могла пояснити, чому виконала саме цей шлях.
```

Тому Ordo ближча не до “формату документа”, а до “мови процесу”.

## 3.2. Чим Ordo відрізняється від звичайної інструкції

Звичайна інструкція часто виглядає так:

```text
Спочатку визнач тип задачі. Потім постав уточнювальні питання. Якщо даних достатньо, створи документ. Не вигадуй фактів. Якщо є ризики, покажи їх. Перед фіналом перевір результат.
```

Це зрозумілий текст. Але в ньому не видно, де саме початок, які відповіді дозволені, що вважається достатніми даними, яка перевірка обов’язкова, а яка бажана, і що робити при помилці.

Ordo намагається зробити ці речі явними.

Наприклад:

```yaml
entry:
  id: START
  next_node: N1

nodes:
  - id: N1
    question: "Який тип задачі потрібно виконати?"
    answers:
      A: document_generation
      B: analysis
      C: qa_check
      D: not_sure

state:
  task_type: unknown
  data_sufficiency: unknown
  confirmed_contracts: []
  assumptions: []

gates:
  - id: G1
    rule: task_type_must_be_known
    on_fail: ask_clarifying_question

  - id: G2
    rule: no_unsupported_facts
    on_fail: remove_or_mark_unsupported
```

Це вже не просто “порада моделі”. Це структура, за якою модель має рухатися.

## 3.3. Ordo задає дозволений простір дій

<p><img src="../assets/mascots/64x64/Nebu_attention_64x64.png" alt="Nebu attention" width="64" /></p>

Один із головних принципів Ordo: модель не повинна мати необмежену свободу там, де потрібен контроль.

У звичайному prompt-і модель часто сама вирішує:

```text
- яке питання поставити;
- чи достатньо даних;
- чи можна створювати фінальний результат;
- чи треба питати підтвердження;
- чи можна зробити припущення;
- які правила важливіші.
```

Іноді це корисно. Але у складних процесах це небезпечно.

Ordo задає дозволений простір дій:

```text
зараз дозволено тільки поставити питання N1;
після відповіді A перейти до N2A;
після відповіді B перейти до N2B;
не дозволено генерувати output до проходження G_CONTRACTS_CONFIRMED;
не дозволено називати proposed value confirmed;
не дозволено ховати hard gate у FREEFORM.
```

Це дуже схоже на різницю між розмовою і процесом.

У розмові модель може імпровізувати.

У процесі модель має виконувати правила.

Ordo потрібна саме для другого випадку.

## 3.4. Модель як виконавець, а не автор процесу

У звичайній роботі з AI модель часто стає не тільки виконавцем, а й автором процесу. Користувач дає приблизну задачу, а модель сама вирішує, як її розкласти.

Це зручно для творчих або відкритих задач.

Але якщо процес уже визначений, модель не має його вигадувати заново.

Наприклад, якщо в playbook-у сказано, що guided intake має починатися з вибору типу входу, модель не повинна починати з alias, QA або source row.

В Ordo це формалізується так:

```json
{
  "op": "ENTRY.DEF",
  "id": "START_HISTORY_EVENT_INTAKE",
  "first_node": "N1_ENTRY_SOURCE"
}
```

І далі:

```json
{
  "op": "NODE.DEF",
  "id": "N1_ENTRY_SOURCE",
  "allowed_answers": ["A", "B", "C", "D", "E", "F"],
  "next_policy": "follow_answer_transition_only"
}
```

Тобто модель не проектує процес на льоту. Вона виконує вже описаний процес.

Це одна з найважливіших ідей Ordo.

## 3.5. Людська мова залишається важливою

Може здатися, що Ordo хоче замінити людську мову на технічні структури. Насправді ні.

Людська мова потрібна для опису сенсу.

Наприклад, доменне правило може звучати так:

```text
Не можна вважати приклад готовим контрактом, якщо аналітик або фактичні дані не підтвердили, що саме цей контракт використовується в поточній події.
```

Це важливий зміст. Його можна частково формалізувати:

```json
{
  "op": "GATE.DEF",
  "id": "G_EXAMPLE_NOT_CONTRACT",
  "assert": "EXAMPLES_ARE_NOT_PROMOTED_TO_CONFIRMED_CONTRACTS",
  "on_fail": "BLOCK_AND_REQUEST_EVIDENCE"
}
```

Але пояснення, чому це важливо, може залишатися людською мовою в controlled FREEFORM:

```json
{
  "op": "FREEFORM.ADD",
  "id": "F_EXAMPLE_WARNING",
  "role": "domain_explanation",
  "binding": "GATE.G_EXAMPLE_NOT_CONTRACT",
  "text": "Приклади в playbook показують форму, але не стають контрактом для нової події без підтвердження."
}
```

Ordo не намагається стиснути весь сенс у короткі коди. Вона відділяє те, що має бути виконане як правило, від того, що пояснює правило.

## 3.6. Ordo як проміжний шар між людиною і моделлю

<p><img src="../assets/mascots/64x64/Nebu_interesting_64x64.png" alt="Nebu interesting" width="64" /></p>

У майбутньому робота може виглядати так:

```text
людський опис
  ↓
Ordo Source
  ↓
Semantic JSON IR
  ↓
модель або Ordo-runtime
  ↓
validated result
```

Людина не обов’язково має писати IR вручну. Вона може описати процес людською мовою. Потім compiler або сама модель допоможе перетворити цей опис на Ordo Source.

А вже Ordo Source може бути скомпільований у Semantic JSON IR, який краще підходить для виконання.

Тобто Ordo — це не тільки “мова для людей” і не тільки “мова для машин”. Це міст між ними.

Людині потрібен зрозумілий опис.

Моделі потрібна структурована інструкція.

Runtime-у потрібні вузли, gates, state і outputs.

Ordo намагається об’єднати ці потреби в одному підході.

## 3.7. Ordo не обов’язково потребує окремого runtime вже зараз

Ідеальний майбутній варіант — це Ordo-runtime, який сам читає IR, веде state, визначає current node, запускає gates і дає моделі тільки поточну semantic task.

Але Ordo можна використовувати вже зараз без runtime.

Для цього достатньо дати моделі Ordo-native package і сказати:

```text
Працюй за execution contract.
Перед кожною відповіддю визнач current node, selected docs, state update і gate.
Не використовуй playbook як суцільний markdown.
Не переходь до наступного кроку без дозволеного transition.
```

Це ще не повна автоматизація, але це вже краще, ніж просто прикріпити великий all-in-one документ.

З часом частину цієї роботи можна винести з моделі в helper-runner:

```text
IR читає helper;
state зберігає helper;
next node визначає helper;
модель формує людське питання або документ;
validation перевіряє helper або окремий модуль.
```

У такій архітектурі модель стає менш відповідальною за механіку процесу і більше відповідає за зміст.

## 3.8. Ordo як “програмування без коду” для AI-поведінки

<p><img src="../assets/mascots/64x64/Nebu_thinking_64x64.png" alt="Nebu thinking" width="64" /></p>

Ordo можна назвати програмуванням поведінки AI-моделі.

Але це не програмування в сенсі Java, Python або JavaScript. Тут ми не описуємо, як комп’ютер має рахувати числа або викликати функції.

Ми описуємо, як модель має поводитися:

```text
- коли питати;
- коли мовчати;
- коли генерувати;
- коли перевіряти;
- коли зупинятися;
- коли просити підтвердження;
- коли використовувати FREEFORM;
- коли відмовитися від фіналізації.
```

Це особливий тип програмування.

У класичному коді ми керуємо машиною.

В Ordo ми керуємо моделлю, яка працює з мовою, змістом і невизначеністю.

Тому Ordo не може бути такою ж жорсткою, як звичайна мова програмування. Вона має залишати місце для людського сенсу, доменного тексту і controlled FREEFORM.

Але вона має бути достатньо чіткою, щоб модель не перетворювала складний процес на імпровізацію.

## 3.9. Які задачі добре підходять для Ordo

Ordo особливо корисна, коли задача має структуру.

Наприклад:

```text
- створення аналітичного пакета;
- guided intake;
- QA-сценарії;
- compliance review;
- юридичний аналіз із gates;
- генерація документів за шаблонами;
- перевірка артефактів;
- міграція playbook-а;
- робота з великими knowledge base;
- агентні процеси з human approval.
```

Ordo менш потрібна для дуже простих задач:

```text
- переписати одне речення;
- дати коротке пояснення;
- перекласти одну фразу;
- придумати 10 назв;
- відповісти на просте питання.
```

Хоча навіть просту задачу можна описати в Ordo, не завжди це варто робити.

Головне правило:

```text
Якщо помилка процесу дорожча за вартість формалізації — використовуйте Ordo.
```

## 3.10. Типові помилки в розумінні Ordo

### Помилка 1. “Ordo — це просто JSON”

Ні. JSON — лише один із форматів представлення. Ordo — це модель виконання.

### Помилка 2. “Ordo має формалізувати абсолютно все”

Ні. Якщо формалізація спотворює сенс, використовується controlled FREEFORM.

### Помилка 3. “Ordo потрібна тільки для History Event Playbook”

Ні. History Event — це перший великий приклад. Але конструкції Ordo загальні.

### Помилка 4. “Ordo замінює людину”

Ні. Ordo навпаки краще фіксує, де потрібне людське підтвердження.

### Помилка 5. “Ordo завжди має бути складною”

Ні. Для простої задачі Ordo може бути дуже маленькою. Складність має відповідати ризику задачі.

## 3.11. Короткий підсумок розділу

Ordo — це мова керування поведінкою AI-моделі.

Вона не зводиться до YAML, JSON або op-кодів. Вона описує процес виконання:

```text
entry → node → state → gate → output → validation → handoff
```

Ordo дозволяє:

```text
- обмежити дозволені дії моделі;
- вести state;
- розділити підтверджені дані й припущення;
- виконувати gates;
- використовувати controlled FREEFORM;
- перевіряти rendered artifacts;
- рухатися від людського опису до model-oriented IR.
```

Головна зміна мислення така:

```text
ми не просто просимо модель відповісти;
ми задаємо процес, у межах якого вона має діяти.
```

## Міні-вправа

Візьміть будь-який ваш процес, який ви зараз описуєте prompt-ом.

Наприклад:

```text
Підготуй задачу для Jira на основі мого опису.
```

Спробуйте відповісти:

```text
1. Який entry point процесу?
2. Які 2-3 перші node-и потрібні?
3. Які відповіді користувача треба зберігати у state?
4. Який gate має зупинити модель перед фінальним результатом?
5. Який output має бути створений?
6. Що в цьому процесі може лишитися у FREEFORM?
```

Якщо ви можете відповісти на ці питання, ви вже почали проектувати Ordo-програму.

---

## Керування моделлю не означає заміну моделі

Ordo не намагається прибрати мислення ШІ. Навпаки, ШІ залишається активним: питає, пояснює, радить, інтерпретує відповіді й пропонує рішення. Ordo керує тим, щоб ця свобода не зламала процес: Process Rail тримає state, gates, allowed deviations і resume rules.
