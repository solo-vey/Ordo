# Розділ 39. Ordo як native model language

## Навіщо це потрібно

До цього моменту ми розглядали Ordo у двох практичних режимах.

Перший режим — Ordo без runtime. У цьому випадку Ordo є сильно структурованою інструкцією, яку модель читає як текст і намагається виконувати дисципліновано.

Другий режим — Ordo з helper-runner. Тут частину контролю бере на себе зовнішній виконавчий шар: він зберігає state, перевіряє gates, пише trace, запускає tests і не дозволяє моделі перескочити через критичні зупинки.

Але стратегічна ціль Ordo більша. Ідеальний майбутній режим — це native model support.

Тобто модель не просто читає Ordo як текст, а розуміє її як власну мову виконання.

У такому режимі Ordo стає не “інструкцією для моделі”, а частиною того, як модель організовує своє мислення, виконання, перевірку і пояснення результату.

## Просте пояснення

Звичайний prompt для моделі — це прохання.

Ordo без runtime — це структурований контракт.

Ordo з helper-runner — це контракт із зовнішнім контролером.

Ordo як native model language — це ситуація, коли модель сама має внутрішню підтримку таких конструкцій:

```text
ENTRY.DEF
NODE.DEF
STATE.SCHEMA
GATE.REPORT
ASSERT.NOT
TRACE.LOG
TEST.DEF
IMPROVEMENT.RECORD
FREEFORM.COVERAGE
```

Тобто модель не просто бачить ці слова. Вона знає, що вони означають на рівні execution behavior.

Якщо вона бачить `GATE.REPORT`, вона розуміє, що має оцінити gate.

Якщо вона бачить `ASSERT.NOT`, вона розуміє, що певна дія прямо заборонена.

Якщо вона бачить `STATE.DIFF`, вона розуміє, що треба показати зміну стану.

Якщо вона бачить `EXPECT.NOOP`, вона розуміє, що тест очікує відсутність дії.

## Чим native Ordo відрізняється від prompt-based Ordo

У prompt-based режимі модель може прочитати правило, але все одно пропустити його, якщо текст довгий, неоднозначний або конфліктний.

У native режимі Ordo-конструкції мають бути для моделі спеціальними виконуваними сигналами.

Наприклад, prompt-based модель бачить:

```text
Перед фінальним архівом виконай self-check.
```

І може це пропустити.

Native Ordo-модель бачить:

```yaml
gate:
  id: G_PACKAGE_SELF_CHECK
  method: mechanical
  trust_class: deterministic
  blocking: true
  required_before:
    - OUTPUT.FINAL_ARCHIVE
```

І має сприйняти це не як пораду, а як заблоковану умову переходу.

## Ordo-конструкція

Для native support Ordo має мати чіткий compiled representation.

Наприклад:

```json
{
  "ordo_version": "0.11",
  "execution_mode": "normal",
  "state_schema": {
    "contract_confirmed": "boolean",
    "self_check_passed": "boolean",
    "final_archive_created": "boolean"
  },
  "gates": [
    {
      "op": "GATE.DEF",
      "id": "G_PACKAGE_SELF_CHECK",
      "blocking": true,
      "required_before": ["OUTPUT.FINAL_ARCHIVE"],
      "condition": "self_check_passed == true"
    }
  ],
  "assertions": [
    {
      "op": "ASSERT.NOT",
      "id": "A_NO_ARCHIVE_BEFORE_SELF_CHECK",
      "forbid": "final_archive_created == true AND self_check_passed != true"
    }
  ]
}
```

Для людини це може виглядати як технічний JSON. Але для моделі з native Ordo support це має бути execution map.

## Рівні підтримки Ordo моделлю

Раніше ми вже говорили про можливі рівні підтримки Ordo. Тут зафіксуємо їх у простішій формі.

```text
L0 — модель не знає Ordo;
L1 — модель читає Ordo як структурований текст;
L2 — модель добре розуміє Ordo Source;
L3 — модель виконує Semantic JSON IR як execution contract;
L4 — модель має native Ordo support.
```

На рівні L0 Ordo все ще може бути корисною, бо вона дисциплінує prompt.

На рівні L1 модель краще дотримується структури.

На рівні L2 модель уже розуміє більшість Ordo-патернів.

На рівні L3 модель здатна виконувати IR як карту процесу.

На рівні L4 Ordo стає для моделі власною мовою керування поведінкою.

## Чому це важливо

Native Ordo support дозволив би зменшити головну проблему складних інструкцій: модель не повинна кожного разу “вгадувати”, що важливіше.

Вона має бачити:

```text
це мета;
це контракт;
це state;
це path;
це gate;
це заборонена дія;
це test;
це feedback record;
це output;
це handoff.
```

Тоді складна інструкція перестає бути довгим текстом. Вона стає програмою керування поведінкою.

## Що native Ordo не вирішує

Native Ordo не робить модель абсолютно детермінованою.

Модель усе ще працює зі значеннями, мовою, контекстом, неоднозначністю і неповними даними.

Native Ordo не скасовує людське підтвердження.

Native Ordo не гарантує, що domain rule завжди правильний.

Native Ordo не замінює tests, debug і improvement loop.

Навпаки: що сильніша підтримка Ordo, то важливішими стають tests, trace і version control.

## Типові помилки

Перша помилка — думати, що native Ordo означає “модель більше не помиляється”.

Це неправильно. Native Ordo означає, що помилки стають краще структурованими, видимими і перевірюваними.

Друга помилка — чекати native support і не використовувати Ordo зараз.

Ordo корисна вже зараз як structured instruction language. Native support — це стратегічна ціль, а не умова початку роботи.

Третя помилка — робити Ordo занадто схожою на звичайну мову програмування.

Ordo керує не процесором, а AI-моделлю. Тому вона має враховувати семантику, неоднозначність, контекст, людські підтвердження і пояснюваність.

Четверта помилка — сховати всю складність у compiler.

Навіть якщо compiler стане дуже сильним, людина має розуміти, які contracts, gates, tests і outputs він створює.

## Міні-вправа

Візьміть будь-яку складну інструкцію для AI.

Запитайте себе:

```text
Які частини цієї інструкції модель зараз читає просто як текст?
Які з них мали б бути native Ordo-конструкціями?
Де потрібен gate?
Де потрібен ASSERT.NOT?
Де потрібен STATE.DIFF?
Де потрібен TEST.DEF?
Де потрібен IMPROVEMENT.RECORD?
```

Після цього спробуйте описати, що модель мала б розуміти не як “текст”, а як execution contract.

## Короткий підсумок

Ordo як native model language — це стратегічна ціль, у якій модель підтримує Ordo не як звичайний prompt, а як власну мову керування виконанням.

У такому режимі Ordo-конструкції стають для моделі execution signals: state, gates, assertions, tests, traces, outputs і improvement records.

Це не скасовує людський контроль і не робить AI безпомилковим. Але це переводить складні інструкції з рівня “довгого тексту” на рівень контрольованої, пояснюваної і тестованої поведінки.
