# Розділ 84 — Сліпа автоматизація та причинне покращення playbook

Корисний тест playbook не повинен непомітно підказувати моделі правильну відповідь. У blind RUN модель-виконавець отримує instruction-пакет, контрольований Driver і лише ті факти сценарію, які дозволено відкрити в поточній взаємодії. Очікувані оцінки, еталонні відповіді, результати попередніх RUN і висновки розробників залишаються поза контекстом моделі.

Ordo підтримує дві робочі схеми. `step_bound` Driver проходить реальну канонічну послідовність і відкриває факти за стабільними кроками. `semantic_adaptive` Driver дозволяє моделі самій визначати порядок запитань, класифікує їх у нейтральні intents і повертає тільки мінімально релевантні факти. Це важливо для all-in-one та історично накопичених інструкцій: вигаданий прихований порядок тестував би сценарій Driver, а не самі інструкції.

Завершення процесу не дорівнює якості. Driver може підтвердити проходження protocol gates, але незалежний evaluator повинен окремо оцінити виконання процесу і створені документи. Документ може бути структурно повним, але загальним, суперечливим або непридатним до роботи.

Найсильніша практика покращення починається після слабкого blind RUN. Замість здогадок розробники ставлять тій самій моделі-виконавцю вузький causal diagnostic prompt: який node, prompt, template, contract, набір фактів і gates створили конкретний проблемний елемент? Відповідь моделі є evidence про заявлений execution path, але не абсолютною істиною. Її потрібно звірити з trace, логами та файлами пакета.

Після цього змінюють найвужчий відповідальний компонент, збирають новий blind package і повторюють RUN у чистому контексті. Порівняння двох RUN стає regression evidence. Так покращення playbook перетворюється з широкого переписування prompt-ів на контрольований інженерний цикл.
