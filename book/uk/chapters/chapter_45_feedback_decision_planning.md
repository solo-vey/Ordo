# Розділ 45. Feedback decision planning

Після зовнішнього ревʼю важливо не кидатися одразу переписувати код. Для Ordo важливіше інше: кожне зауваження має пройти через коротке рішення.

```text
зауваження → класифікація → рішення → milestone → критерій прийняття → перевірка
```

## Чому це окремий шар

Feedback intake збирає зауваження. Але сам факт, що зауваження існує, ще не означає, що його треба негайно вносити в release candidate.

M50 додає просте правило: спочатку рішення, потім зміна.

## Типи рішень

```text
accepted_now
accepted_next_milestone
deferred
rejected
needs_reproduction
needs_owner_decision
```

Це дозволяє не змішувати blocker, корисну ідею, документаційне уточнення і стратегічне рішення власника.

## Що має бути в рішенні

Кожне рішення має містити:

- короткий опис;
- severity;
- affected layer;
- target milestone;
- rationale;
- acceptance criteria;
- validation commands.

## Як це працює для Ordo

Ordo не просто приймає фідбек. Ordo переводить фідбек у керований change plan. Це продовження тієї ж ідеї Process Rail: модель може мислити гнучко, але процес не має втрачати трасу.

## Важливе обмеження

M50 не змінює CLI, runtime або бізнесову логіку пакетів. Це decision layer для наступних змін.
