# Розділ 2. Що таке Ordo простими словами

## 2.1. Ordo — це не ще один “формат prompt-а”

Коли люди вперше чують про Ordo, легко подумати, що це просто ще один спосіб оформлювати prompt-и: трохи YAML, трохи JSON, кілька назв на кшталт `intent`, `gate`, `state`, `result`.

Але це не зовсім так.

Prompt — це звернення до моделі.

Ordo — це опис процесу, який модель має виконати.

Prompt часто звучить так:

```text
Зроби мені результат.
```

Ordo звучить інакше:

```text
Ось мета.
Ось контракт.
Ось що вже відомо.
Ось які питання можна ставити.
Ось у якому порядку рухатися.
Ось що заборонено.
Ось де потрібно зупинитися.
Ось які перевірки пройти.
Ось який результат можна видати.
```

Тобто Ordo не просто просить модель щось зробити. Ordo задає рамки виконання.

Можна сказати так:

<p><img src="../assets/mascots/64x64/Nebu_idea_64x64.png" alt="Nebu idea" width="64" /></p>

```text
Prompt — це прохання.
Ordo — це інструкційна програма.
```

Або ще простіше:

```text
Prompt каже моделі: “зроби”.
Ordo каже моделі: “виконай процес”.
```

## 2.2. Побутова аналогія: рецепт і кухня

Уявімо, що ви просите людину приготувати борщ.

Можна сказати:

```text
Приготуй борщ.
```

Якщо людина досвідчена, вона, можливо, зробить усе добре. Але результат залежить від її пам’яті, звичок, смаку і припущень.

Можна дати рецепт:

```text
1. Підготуй інгредієнти.
2. Звари бульйон.
3. Додай буряк.
4. Додай капусту.
5. Перевір сіль.
6. Подай зі сметаною.
```

Це вже краще. Але навіть рецепт може бути нечітким. Наприклад, що робити, якщо немає буряка? Чи можна замінити інгредієнт? Коли страва вважається готовою? Хто підтверджує смак?

Ordo — це не просто рецепт. Це рецепт із правилами виконання:

```text
- якщо інгредієнта немає — не вигадуй заміну, а запитай;
- не переходь до подачі, поки не перевірено готовність;
- якщо страва пересолена — не видавай як finished;
- якщо користувач просив вегетаріанський варіант — не використовуй м’ясний бульйон;
- у кінці покажи, які перевірки пройдені.
```

Саме так Ordo працює з AI-моделлю.

Вона не просто описує бажаний результат. Вона описує, як модель має рухатися до результату.

## 2.3. Ordo як карта маршруту

Ще одна проста аналогія — навігатор.

Prompt часто схожий на фразу:

```text
Доїдь до Львова.
```

Модель може вибрати маршрут сама. Іноді це добре. Але якщо вам важливо не заїжджати на платні дороги, не їхати через певну область, зробити зупинку, перевірити паливо і не завершити маршрут без підтвердження, одного речення вже недостатньо.

Ordo — це маршрут із контрольними точками:

```text
start
→ обрати напрям
→ перевірити обмеження
→ пройти checkpoint
→ якщо дорога закрита — перейти на альтернативний шлях
→ не завершувати маршрут без arrival confirmation
```

У термінах Ordo це виглядає так:

```text
ENTRY.DEF
→ NODE.DEF
→ STATE.UPDATE
→ GATE.CHECK
→ PATH.SELECT
→ OUTPUT.DEF
→ HANDOFF
```

Тобто Ordo дає моделі не тільки “куди їхати”, а й “як саме рухатися”.

## 2.4. Головні частини Ordo

У найпростішому вигляді Ordo складається з кількох базових частин.

### Intent

`Intent` — це мета.

Наприклад:

```yaml
intent:
  goal: create_summary
```

Або:

```yaml
intent:
  goal: guide_user_through_history_event_intake
```

Intent відповідає на питання:

```text
Навіщо взагалі запускається цей процес?
```

### Contract

`Contract` — це домовленість про результат і правила.

Наприклад:

```yaml
contract:
  output:
    format: bullet_list
    max_items: 3
  rules:
    - do_not_invent_facts
    - use_only_input_text
```

Contract відповідає на питання:

```text
Що вважається правильним результатом?
Що заборонено?
Які обмеження треба виконати?
```

### Context

`Context` — це вхідні дані, документи, джерела або попередні рішення.

Наприклад:

```yaml
context:
  source_text: "$USER_INPUT.text"
  playbook: "history_event_playbook_ordo_v0_10"
```

Context відповідає на питання:

```text
Звідки модель бере дані?
```

### State

`State` — це поточний стан процесу.

Наприклад:

```yaml
state:
  current_node: N1
  selected_path: null
  confirmed_contracts: []
  open_questions: []
```

State відповідає на питання:

```text
Що вже відомо?
Де ми зараз?
Що ще не підтверджено?
```

### Path

`Path` — це обраний маршрут виконання.

Наприклад:

```yaml
path:
  selected: Path_1
  reason: "internal source row change"
```

Path відповідає на питання:

```text
Яким сценарієм ідемо?
```

### Step

`Step` — це конкретна дія.

Наприклад:

```yaml
step:
  id: S1
  do: read_input_text
```

Step відповідає на питання:

```text
Що модель робить зараз?
```

### Gate

`Gate` — це контрольна точка.

Наприклад:

```yaml
gate:
  id: G1
  method: self_verification
  trust_class: model_judgment
  check: no_unsupported_facts
  on_fail: block_or_repair
```

Gate відповідає на питання:

```text
Чи можна йти далі?
```

### Result

`Result` — це те, що створюється.

Наприклад:

```yaml
result:
  type: summary
  status: ready
```

Result відповідає на питання:

```text
Що отримує користувач?
```

### Handoff

`Handoff` — це передача результату.

Наприклад:

```yaml
handoff:
  include:
    - result
    - status
    - gate_report
```

Handoff відповідає на питання:

```text
Що саме віддати користувачу або передати далі?
```

## 2.5. Ordo не змушує все робити машинним кодом

<p><img src="../assets/mascots/64x64/Nebu_attention_64x64.png" alt="Nebu attention" width="64" /></p>

Важливо: Ordo не каже, що всю людську інструкцію потрібно перетворити на короткі op-коди.

Це була б помилка.

У реальних процесах є багато речей, які краще залишити людською мовою:

```text
- доменні пояснення;
- приклади;
- стилістичні вимоги;
- довгі шаблони документів;
- застереження;
- історичні примітки;
- пояснення для аналітика.
```

Для цього в Ordo є controlled `FREEFORM`.

Але різниця в тому, що `FREEFORM` не є хаотичним текстом. Він має роль, прив’язку і причину.

Наприклад:

```json
{
  "op": "FREEFORM.ADD",
  "id": "F1",
  "role": "domain_guidance",
  "binding": "Path_5",
  "reason": "domain explanation cannot be safely reduced to core opcodes",
  "text": "For Path 5, the first received state creates only a baseline snapshot by default."
}
```

Тобто Ordo не викидає людський текст. Вона ставить його на правильне місце.

## 2.6. Ordo має кілька рівнів читання

Ordo можна читати на різних рівнях.

### Людський рівень

Це пояснення, яке читає аналітик або автор playbook-а.

Наприклад:

```text
Спочатку потрібно визначити, звідки система має зрозуміти, що історична подія сталася.
```

### Ordo Source

Це структурований опис, який ще зручний для людини.

Наприклад:

```yaml
entry:
  id: START_HISTORY_EVENT_INTAKE
  first_node: N1
```

### Semantic JSON IR

Це compiled representation, краще придатне для моделі або runtime.

Наприклад:

```json
{
  "op": "ENTRY.DEF",
  "id": "START_HISTORY_EVENT_INTAKE",
  "first_node": "N1"
}
```

### Compact IR

Це майбутній компактний машинний формат.

Наприклад:

```json
["ENTRY", "START_HISTORY_EVENT_INTAKE", "N1"]
```

Усі ці рівні описують одне й те саме, але для різних читачів.

Людині зручніше читати пояснення й YAML.

Моделі або runtime зручніше працювати з JSON IR.

## 2.7. Ordo не є заміною моделі

Ordo не виконує роботу замість AI-моделі.

Модель усе ще потрібна для:

```text
- розуміння людського тексту;
- формулювання відповідей;
- заповнення документів;
- узагальнення;
- пояснення;
- роботи з неоднозначностями;
- пропозицій і покращень.
```

Але Ordo обмежує і направляє модель.

Можна сказати так:

```text
Модель генерує зміст.
Ordo керує виконанням.
```

Або:

```text
Модель — це двигун.
Ordo — це маршрут, правила руху і контрольні пункти.
```

Без двигуна маршрут нікуди не поїде. Але без маршруту двигун може поїхати не туди.

## 2.8. Ordo не є жорсткою бюрократією

<p><img src="../assets/mascots/64x64/Nebu_thinking_64x64.png" alt="Nebu thinking" width="64" /></p>

Ще одне можливе непорозуміння: Ordo може здатися надто формальною.

Навіщо стільки структури? Чому не можна просто написати нормальний prompt?

Можна. Для простих задач — краще так і робити.

Ordo потрібна там, де ціна помилки вища:

```text
- складні документи;
- багато правил;
- багато кроків;
- треба ставити питання;
- треба відрізняти candidate від confirmed;
- треба не вигадувати;
- треба створити пакет артефактів;
- треба перевірити результат;
- треба мати пояснюваний trace.
```

Ordo — це не про формальність заради формальності.

Ordo — це про керованість.

## 2.9. Малий приклад: звичайне завдання

Користувач пише:

```text
Підсумуй цей текст українською у максимум 3 пункти.
Не додавай фактів, яких немає в тексті.
```

Звичайна модель може просто відповісти.

Ordo-модель спочатку бачить структуру:

```yaml
intent:
  goal: summarize_text
  language: uk

contract:
  output:
    format: bullet_list
    max_items: 3
  rules:
    - use_only_input_text
    - no_unsupported_facts

gates:
  - input_text_present
  - max_3_items
  - no_unsupported_facts
  - output_language_uk
```

Для користувача результат може виглядати майже так само, як звичайна відповідь.

Але внутрішньо процес більш контрольований.

## 2.10. Великий приклад: playbook

<p><img src="../assets/mascots/64x64/Nebu_interesting_64x64.png" alt="Nebu interesting" width="64" /></p>

Користувач пише:

```text
Хочу створити нову історичну подію.
```

Звичайна модель може почати пропонувати alias або source.

Ordo-модель бачить:

```json
{
  "op": "ENTRY.DEF",
  "id": "START_HISTORY_EVENT_INTAKE",
  "first_node": "N1"
}
```

Потім:

```json
{
  "op": "NODE.DEF",
  "id": "N1",
  "question": "Звідки система має зрозуміти, що історична подія сталася?",
  "answers": ["A", "B", "C", "D", "E", "F"]
}
```

І тільки після відповіді користувача вона рухається далі.

Це і є різниця між “прочитати playbook” і “виконувати playbook”.

## 2.11. Ordo як дисципліна мислення

Навіть якщо у вас немає Ordo-runtime, Ordo вже корисна як спосіб мислення.

Вона змушує автора інструкції запитати себе:

```text
- Яка мета?
- Який контракт?
- Який стан процесу?
- Які дозволені відповіді?
- Які gates?
- Які outputs?
- Що не можна робити?
- Що лишається у FREEFORM?
- Що потрібно підтвердити людиною?
```

Ці питання самі по собі роблять інструкції кращими.

Навіть якщо модель не підтримує Ordo нативно, Ordo-native playbook значно краще організовує роботу, ніж великий хаотичний all-in-one prompt.

## 2.12. Короткий підсумок розділу

Ordo — це мова керування виконанням AI-моделі.

Вона не є просто YAML або JSON.

Вона описує:

```text
- мету;
- контракт;
- контекст;
- стан;
- дерево рішень;
- кроки;
- перевірки;
- результати;
- передачу в роботу;
- контрольований freeform.
```

Найкоротше визначення:

```text
Ordo — це спосіб перетворити людську інструкцію на керований execution contract для AI-моделі.
```

Prompt просить модель щось зробити.

Ordo задає процес, у межах якого модель має це зробити.

## Міні-вправа

Візьміть просту задачу:

```text
Підготуй коротку відповідь клієнту про затримку доставки.
```

Спробуйте розкласти її в Ordo-стилі:

```text
1. Який intent?
2. Який contract результату?
3. Який context потрібен?
4. Який state треба вести?
5. Які gates потрібні?
6. Який output має бути?
7. Що не можна вигадувати?
```

Навіть така проста вправа показує головне: Ordo починається не з формату, а з мислення про контрольоване виконання.

---

## Ordo після Process Rail reframing

Найпростіше пояснення: Ordo дозволяє ШІ вести складний процес людською мовою, але не дає йому загубити маршрут. Для цього Ordo описує Process Rail: що вже зібрано, який крок активний, які gates ще не пройдені, що робити після уточнення або зміни попередньої відповіді.
