# ET-9 — Regression and Pilot Report

## Scope

Validate the APF Execution Trace integration without enabling trace capture for APF itself.

## Scenario A — generated playbook with trace disabled

Input: `trace_disabled_playbook_config.yaml`.

Expected and observed:

- trace capability remains declared;
- runtime capture is disabled;
- no actual `EXECUTION_TRACE` instance is generated by APF;
- runtime toggle remains available;
- existing traces, if any, are not deleted;
- release is not blocked solely because capture is disabled.

Result: **passed**.

## Scenario B — generated playbook with trace enabled

Inputs:

- `trace_enabled_playbook_config.yaml`;
- `trace_enabled_event_mapping.yaml`;
- `trace_enabled_adapter_conformance.yaml`.

Expected and observed:

- canonical configuration is present;
- standard capture level is used;
- required lifecycle, gate, state and output event coverage is present;
- `runtime_enforced` has passing adapter conformance evidence;
- replay readiness fields are present;
- APF produces contracts and package artifacts only, not a real runtime trace.

Result: **passed**.

## Regression boundary

Confirmed:

- APF internal trace remains disabled by default;
- mini-prompt support remains unchanged;
- generated playbooks can explicitly disable trace capture;
- enabled trace requires the expected artifact and adapter evidence;
- no hidden chain of thought is requested or stored;
- Ordo core is not modified by APF.

## Outcome

ET-9 status: **passed**  
Checks passed: **24**  
Blocking issues: **0**

The Execution Trace enhancement is ready for ET-10 release-candidate assembly.
