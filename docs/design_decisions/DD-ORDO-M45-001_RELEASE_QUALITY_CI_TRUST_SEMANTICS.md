# DD-ORDO-M45-001 — Release Quality, CI and Trust Semantics Fix

## Status

Accepted for M45.

## Context

External usability audit of the M43/M44 lean workspace found release-quality defects that would break CI or weaken Ordo's trust guarantees:

- GitHub Actions referenced a removed legacy package.
- Generated package metadata could remain in the source archive while validation reports claimed it was removed.
- `self_consistency` gates were not linted against `repeated_model_judgment`.
- `ordo test` reported `passed` without visibly saying that the runner is static.
- Compiler-generated Process Rail op-codes were not all present in `language/registry/OPCODE_CATALOG.md`.

## Decision

M45 turns these audit findings into deterministic repository and language checks:

1. GitHub Actions now validates only active reference packages.
2. A repository-level `ordo repo-check` command validates workflow package paths and absence of generated metadata.
3. `compile` checks generated IR op-codes against `language/registry/OPCODE_CATALOG.md`.
4. The opcode catalog now includes Process Rail / hybrid execution op-codes generated by the compiler.
5. The linter warns when `self_verification` and `self_consistency` gates use the wrong trust class.
6. `packages/ordo_hybrid_executor` now uses `self_consistency + repeated_model_judgment`.
7. `ordo test` explicitly prints `[static mode]` and reports assertion behavioral-evaluation coverage.
8. Runtime assertion results now distinguish `behavioral` from `structural_only` evaluation.

## Consequences

- CI no longer references removed packages.
- Generated metadata is treated as a source-archive violation.
- Trust-class drift becomes visible during lint.
- Future compiler op-code additions must be reflected in the canonical opcode catalog.
- CLI output is more honest about static vs behavioral validation.
