# Blind Automation and Causal Improvement Guide

## Choosing a profile

Choose `step_bound` only when the canonical process has a stable mandatory sequence. Choose `semantic_adaptive` when the model legitimately decides question order. Do not use a step Driver to make an unstructured instruction set appear deterministic.

## Building the package

Declare the profile in README, isolate hidden facts, configure minimal disclosure, bind Driver identity and fixture checksum, include terminal contracts and audit logging, provide independent process/document rubrics, and prohibit self-scoring in the launch prompt.

## Diagnosing a weak result

Start from an independently identified artifact defect. Ask for causal reconstruction of that element only. Compare the model report with execution trace, loaded files, fixture disclosures, and gate outcomes. Patch the narrowest component supported by evidence.

## Regression closure

Create a new package identity, run in a clean model context, preserve both runs, compare the affected rubric dimensions, and verify that unrelated outputs did not regress. A better terminal state alone is insufficient.

## Safety rules

Never expose expected answers before execution, never transfer approvals to changed artifact versions, never score the model using harness-capture quality, never treat structural presence checks as semantic quality, and never accept a model's diagnostic narrative without corroboration.
