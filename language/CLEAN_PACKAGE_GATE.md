# Clean Package Gate

Status: `M67.4 aligned to implemented CLI`

`clean_package_gate` is the language-level gate that interprets the deterministic output of `ordo clean-check` for package handoff, release review, or runtime-readiness review.

The gate is not a runtime opcode and does not execute package logic. It is a release/packaging consistency gate over declared package files, manifests, prompts, startup entries, derived artifacts, and delta backlog records.

## Purpose

A package is clean when the declared package surfaces agree with each other:

- `ordo.yml` identifies the package and its source YAML;
- declared source YAML parses;
- declared manifest paths exist;
- declared checksums match when checksums are present;
- `prompt_refs` resolve when `prompt_registry` exists;
- startup entry files exist when `startup_package_profile` exists;
- derived artifacts are either current enough to be used or explicitly covered by `delta_backlog`;
- open blockers in `delta_backlog` are not silently expired or ownerless under the selected profile.

Clean does not mean that the package is business-complete or that human decisions are approved. It means package consistency is sufficient for the selected review profile.

## CLI alignment

The canonical implementation surface is:

```bash
ordo clean-check <package>
ordo clean-check <package> --profile light|standard|strict
ordo clean-check <package> --json
ordo clean-check <package> --fail-on-warning
ordo clean-check <package> --out reports/clean_check_report.json
```

`clean_package_gate` consumes the report produced by this command. The report mode is `clean_package_check` and the status values are:

- `passed`
- `passed_with_warnings`
- `blocked`
- `not_applicable`

## Minimal gate mapping

| CLI status | Gate decision | Meaning |
|---|---|---|
| `passed` | `approved` | No blocking errors and no warnings under the selected profile. |
| `passed_with_warnings` | `approved_with_warnings` | No blocking errors, but warnings remain. |
| `blocked` | `blocked` | At least one blocking error exists. |
| `not_applicable` | `not_applicable` | Target is not suitable for clean package review. |

If `--fail-on-warning` is used, a `passed_with_warnings` report may still be interpreted as a failing CI result, while the semantic gate decision remains `approved_with_warnings`.

## Profiles

| Profile | Use |
|---|---|
| `light` | draft authoring and fast review; some drift is warning-level. |
| `standard` | normal package handoff; unresolved declared files/checksums and unbacklogged stale derived artifacts block. |
| `strict` | release/stable review; ownerless or expired blockers and unresolved warnings should fail the handoff. |

## v1 check groups

The implemented M67.3 CLI provides these deterministic v1 checks:

- `package_root_exists`
- `package_manifest_present`
- `package_manifest_parse`
- `source_yaml_declared`
- `source_yaml_exists`
- `source_yaml_parse`
- `declared_manifest_paths_exist`
- `declared_manifest_checksums_match`
- `prompt_registry_refs_resolve_when_present`
- `startup_profile_entries_exist_when_present`
- `derived_artifacts_current_or_backlogged_when_declared`
- `delta_backlog_blockers_not_expired_when_declared`

Future versions may add additional checks, but they must not change runtime semantics or promote packaging conventions into opcodes.

## Authority boundary

`clean_package_gate` may:

- block handoff/release when declared package consistency fails;
- record warnings and errors;
- require human review for warnings or known deltas;
- require regeneration or refresh by a separate explicit command.

`clean_package_gate` must not:

- regenerate runtime artifacts by itself;
- regenerate lockfiles by itself;
- rebuild embedded CLI bundles by itself;
- mutate source YAML;
- modify prompts;
- approve business content;
- bypass gates;
- confirm state;
- choose branches;
- interpret human-language prompt authority deterministically;
- add opcodes or runtime semantics.

## Recommended placement

A package may declare clean gate expectations in a package profile, release profile, or future validation profile, but the gate is evaluated by CLI evidence rather than by trust in documentation.

Minimum evidence:

```text
reports/clean_check_report.json
```

The report should be generated by `ordo clean-check` and included in handoff/release evidence when clean package readiness is claimed.
