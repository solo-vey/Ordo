# M82.0 — Runtime Execution Threat Model and Process-Boundary Contract

## Status

`contract-defined / implementation-not-yet-accepted`

Backlog item: `BL-ORDO-003`

## Purpose

Define the mandatory safety and evidence contract for executing generated PathWalk real-module testcases before further runner implementation or runtime integration.

This milestone changes no Ordo language semantics and makes no scoring or calibration claim.

## Baseline and reconciliation

M76.2 provides a working prototype with plan/job/collect commands, child-process execution, a watchdog, sandbox copying and raw result artifacts. It is retained as implementation evidence.

BL-ORDO-003 remains open because the current prototype does not yet prove the complete required failure taxonomy, malformed-output handling, cleanup guarantees, path/input trust boundaries, cross-platform behavior, or regression matrix required for closure.

Therefore:

- M76.2 is not reverted;
- M82.0 supersedes its incomplete safety contract, not its code;
- M82.1–M82.4 must harden and validate the implementation against this contract.

## Protected assets

The execution boundary must protect:

- the parent PathWalk/Ordo process;
- the canonical source workspace;
- files outside the assigned sandbox;
- host CPU, memory, file descriptors and disk capacity;
- secrets and inherited environment variables;
- other testcase jobs;
- integrity of raw execution evidence;
- correctness of later scoring and calibration claims.

## Threat model

Generated testcase execution is treated as untrusted or faulty input even when generated by trusted tooling.

Threats in scope:

1. infinite loop or excessive runtime;
2. child process crash or abnormal signal termination;
3. runaway subprocess descendants;
4. excessive stdout or stderr;
5. excessive file creation or file size;
6. excessive CPU, memory or open-file use;
7. attempts to read or modify files outside the sandbox;
8. path traversal or unsafe absolute paths in plan/job descriptors;
9. malformed, missing, truncated or contradictory runtime output;
10. partial result writes caused by interruption;
11. environment-variable or stdin dependence;
12. one testcase contaminating another;
13. collector accidentally executing or repairing jobs;
14. parent silently converting failures into passes;
15. scoring or calibration consuming unstable evidence.

Out of scope for this milestone:

- hostile kernel-level containment escape;
- full container/VM sandboxing;
- network policy enforcement where the host offers no isolation primitive;
- execution of arbitrary third-party binaries supplied by a testcase;
- scoring, calibration or production recommendation.

These out-of-scope items must not be implied as solved.

## Trust boundaries

### Trusted

- reviewed PathWalk planner and collector code;
- reviewed Ordo CLI/runtime entry point;
- immutable execution contract and schemas;
- canonical testcase package after validation.

### Untrusted or conditionally trusted

- generated testcase content;
- testcase answers and scripted steps;
- runtime stdout/stderr;
- runtime report files;
- child exit code and termination behavior;
- any path originating from an external plan or testcase package;
- inherited host environment.

## Mandatory process boundary

The execution flow must be:

`validated source/testcase artifacts → immutable plan → one job descriptor → one isolated child process → atomic raw result artifact → collect-only parent`

Mandatory properties:

1. Planner never executes a testcase.
2. Collector never executes or retries a testcase.
3. Every job executes in a separate short-lived process group.
4. Every job has a unique sandbox created from validated inputs.
5. Canonical source artifacts are copied read-only in intent; execution must not use them in place.
6. Child stdin is closed.
7. Environment is allowlisted or explicitly minimized.
8. A hard wall-clock watchdog applies to each job.
9. Timeout termination targets the complete child process group.
10. Resource limits are applied where supported and their availability is recorded.
11. stdout and stderr are bounded without changing the child exit status.
12. Results are written atomically: temporary file then rename.
13. Existing result artifacts are not overwritten unless an explicit force/retry policy is invoked.
14. Sandbox cleanup follows an explicit retention policy and is reported.
15. Parent and collector never infer success from missing evidence.

## Input validation contract

Before a job starts, the worker must validate:

- supported plan schema version;
- unique job ID and case ID;
- job belongs to the referenced plan;
- source, case, result and sandbox paths resolve under approved roots;
- no `..` traversal or unapproved symlink escape;
- source and case files exist and are regular files;
- timeout and output limits are within configured bounds;
- testcase structure contains the required fields for its case type;
- result path does not alias canonical source or testcase files.

Validation failure produces an infrastructure/input failure result; the runtime must not be started.

## Evidence contract

Each attempted job must produce one result artifact containing at least:

- schema version;
- plan ID/fingerprint;
- job ID and case ID;
- source and testcase fingerprints;
- normalized status and failure category;
- child exit code or terminating signal;
- timeout flag;
- start/end timestamps and duration;
- watchdog action;
- applied/unsupported resource limits;
- bounded stdout/stderr plus truncation flags;
- runtime report presence, path and parse status;
- sandbox path and cleanup/retention status;
- command identity without secrets;
- explicit `scoring_performed=false` and `calibration_performed=false`.

Raw evidence must not be silently repaired, normalized into a pass, or replaced by model-generated interpretation.

## Required status taxonomy

M82.2 will define the schema in detail. M82.0 fixes the minimum semantic categories:

- `passed` — exit code is successful and required runtime evidence is present and valid;
- `testcase_failed` — runtime completed normally but the testcase/runtime reported a valid negative result;
- `process_failed` — non-zero exit or abnormal termination without valid expected failure evidence;
- `timed_out` — watchdog deadline exceeded;
- `invalid_output` — process completed but required output is missing, malformed, truncated beyond usability or schema-invalid;
- `input_rejected` — plan, path, source or testcase validation failed before execution;
- `infrastructure_error` — runner could not create sandbox, spawn, monitor, terminate or persist evidence reliably;
- `cancelled` — explicit external cancellation, when implemented.

Unknown or missing state must never map to `passed`.

## Cleanup and retention contract

- Each job receives a unique sandbox.
- A passed job may be removed after result persistence according to configured policy.
- Failed, timed-out, invalid-output and infrastructure-error sandboxes are retained by default for diagnosis, unless data policy requires deletion.
- Cleanup outcome is part of the result evidence.
- Cleanup failure cannot change a failed testcase into a pass; it is recorded as an additional infrastructure finding.
- No job may delete paths outside its approved execution root.

## Collector contract

The collector may only:

- read the immutable plan;
- read completed atomic result artifacts;
- validate result schema and plan/job linkage;
- count statuses and missing jobs;
- emit an aggregate summary.

The collector must not:

- run, retry or terminate jobs;
- parse stdout to manufacture missing runtime reports;
- change child status;
- repair malformed evidence;
- score or calibrate testcases;
- claim stability when jobs are missing or invalid.

## Security and portability requirements

- POSIX process-group termination is the reference behavior.
- Non-POSIX platforms must implement an equivalent descendant-termination strategy or report the capability as unsupported and block a full safety claim.
- Optional OS limits must be capability-reported, not silently assumed.
- Network isolation is not claimed unless implemented and tested explicitly.
- Secrets must not be copied into command/result artifacts.

## Acceptance gates for later milestones

### M82.1

- validated path boundary;
- isolated child and process-group watchdog;
- atomic result persistence;
- environment/stdin controls;
- explicit cleanup policy.

### M82.2

- versioned evidence schema;
- complete status/failure taxonomy;
- truncation, signal, capability and cleanup fields;
- collector schema validation.

### M82.3

- PathWalk generated clean and noise cases execute through this boundary;
- generate-only remains available;
- no direct in-process execution path is introduced;
- no scoring/calibration claim.

### M82.4

Regression matrix must cover at least:

- normal completion;
- valid testcase failure;
- non-zero exit;
- signal/crash;
- timeout with descendant process;
- malformed runtime report;
- missing runtime report;
- oversized stdout/stderr;
- invalid/traversing path;
- sandbox creation failure;
- result write failure;
- cleanup success and cleanup failure;
- missing result during collection;
- cross-job isolation;
- existing PathWalk generation and Ordo runtime regressions.

## Definition of done for M82.0

M82.0 is complete when:

- the threat model is explicit;
- trust boundaries are explicit;
- process, input, evidence, cleanup and collector contracts are fixed;
- M76.2 prototype status is reconciled without false closure;
- later milestone acceptance gates are defined;
- no code or production-readiness claim is made.

## Change impact

Expected level: `L3 — shared/runtime utility layer`.

No Ordo language semantic change is authorized by this contract. Implementation changes will be restricted to PathWalk execution tooling and its interfaces with the existing Ordo CLI/runtime.
