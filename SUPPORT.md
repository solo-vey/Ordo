# Support

Use the route that matches the request. Public channels must not contain secrets, private evidence, confidential data, unredacted personal information, or suspected vulnerability details.

## Security vulnerabilities

Use GitHub private vulnerability reporting as described in [`SECURITY.md`](SECURITY.md).

Do not create a public issue for a suspected vulnerability.

## Bugs

Use the GitHub bug-report form for reproducible defects.

Before filing:

1. confirm the behavior on the current `main` branch or identify the exact packaged release;
2. search existing issues;
3. reduce the report to a minimal reproducible case;
4. remove credentials, private data, and confidential artifacts;
5. include commands, expected behavior, actual behavior, and relevant logs.

## Documentation problems

Use the documentation-issue form for missing, incorrect, conflicting, or unclear current documentation.

Historical transfer materials and archived milestone reports are not the current documentation front door.

## Feature proposals

Use the feature-proposal form. Explain the problem, users affected, desired outcome, compatibility impact, and evidence for the need.

A proposal does not imply roadmap commitment or acceptance.

## Usage questions

Review these documents first:

- [`README.md`](README.md)
- [`docs/QUICKSTART.md`](docs/QUICKSTART.md)
- [`docs/README.md`](docs/README.md)
- [`cli/README.md`](cli/README.md)

When a public issue is necessary, provide a minimal non-sensitive example and state which documentation was already checked.

## Conduct concerns

Do not put sensitive conduct reports in public issues.

Use GitHub private vulnerability reporting as the currently approved private repository channel and begin the report title with `Conduct concern:`. Maintainers will treat the report as a private conduct matter rather than a vulnerability advisory.

## Commercial licensing

Review [`COMMERCIAL_LICENSE.md`](COMMERCIAL_LICENSE.md). The public issue tracker is not a commitment to provide commercial support, consulting, custom development, or licensing terms.

## Support boundaries

This public repository is an active release-candidate project. Support is best effort. Maintainers may close requests that are:

- duplicates;
- unsupported historical versions;
- missing a reproducible case;
- outside project scope;
- requests for private data analysis;
- attempts to bypass licensing, safety, evidence, or release-integrity controls.
